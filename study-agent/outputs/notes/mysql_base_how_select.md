# MySQL `SELECT` 语句执行流程笔记

本文档提供了关于 MySQL 内部执行 `SELECT` 查询语句全流程的结构化笔记。

## 1. MySQL 整体架构

### 直觉 (Intuition)
把 MySQL 想象成一家餐厅。**Server 层**是前厅（服务员、经理、收银员），负责接收点单、检查订单有效性并安排如何制作。**存储引擎层**是后厨（厨师、储藏室），负责实际存储食材（数据），并在前厅要求时提取。

### 例子 (Example)
当你运行 `SELECT * FROM product WHERE id = 1;` 时，Server 层处理网络连接并找出查找 `id = 1` 记录的最佳方案。存储引擎层（如 InnoDB）在它的 B+ 树结构中搜索，获取实际的数据行并交还给 Server 层。

### 解释 (Explanation)
MySQL 共分为两层：
- **Server 层**：负责建立连接、分析和执行 SQL。大部分核心功能都在这里实现，包含连接器、查询缓存、解析器、预处理器、优化器、执行器等模块。内置函数和跨存储引擎的功能（如触发器、视图）也在这里实现。
- **存储引擎层**：负责数据的存储和提取。采用插件式架构，支持 InnoDB、MyISAM、Memory 等。它管理实际的数据文件和索引结构（不同的存储引擎支持不同的索引结构，如 InnoDB 默认使用 B+ 树）。

## 2. 第一步：连接器

### 直觉 (Intuition)
连接器就像高级俱乐部的保安和前台。它检查你的身份证（校验凭证），确保俱乐部没有客满（最大连接数限制），并记住你的 VIP 等级（权限），以便决定你后续在俱乐部里能做什么。

### 例子 (Example)
当你执行命令 `mysql -h 127.0.0.1 -u root -p` 时，连接器会先进行 TCP 三次握手，然后验证密码，如果通过，就会加载该用户的权限并缓存起来。

### 解释 (Explanation)
- **建立连接**：通过 TCP 三次握手建立连接。
- **身份验证**：校验用户名和密码，错误则拒绝访问。
- **权限获取**：读取并缓存用户权限。后续该连接里的所有操作都依赖这个一开始读取的权限状态。因此，如果管理员中途修改了该用户的权限，不会影响已经存在的连接，只有新建连接才会生效。
- **连接管理**：
  - 推荐使用**长连接**以减少频繁握手和挥手的开销。
  - 空闲连接超时（由 `wait_timeout` 控制，默认 8 小时）后会自动断开。
  - 最大连接数受 `max_connections` 限制。
  - *长连接内存问题*：长连接可能导致 MySQL 内存占用过多。解决方案是**定期断开长连接**或在 MySQL 5.7+ 版本中使用 `mysql_reset_connection()` 接口重置连接状态。

## 3. 第二步：查询缓存

### 直觉 (Intuition)
查询缓存就像是一份最近问过的问题的标准答案作弊表。如果有人一字不差地问了同样的问题，你直接念作弊表上的答案，不用重新思考计算。

### 例子 (Example)
查询 A: `SELECT * FROM product WHERE id = 1;` -> 缓存未命中，执行并把结果存入缓存。
查询 B: 完全一样的 SQL `SELECT * FROM product WHERE id = 1;` -> 缓存命中，直接返回结果。

### 解释 (Explanation)
- 以 Key-Value 形式在内存中保存（Key 是精确的 SQL 查询语句，Value 是查询结果）。
- **缺陷**：查询缓存非常鸡肋。只要一个表有更新操作，该表相关的所有查询缓存都会被清空。对于更新频繁的表，缓存命中率极低。
- **状态**：在 MySQL 8.0 版本中已被彻底删除。（在旧版本中可以通过设置 `query_cache_type = DEMAND` 来禁用）。

## 4. 第三步：解析器

### 直觉 (Intuition)
解析器是语法老师。它不关心你句子里的主语在现实中存不存在，它只检查你的句子结构是否符合这门语言的语法规则。

### 例子 (Example)
如果你输入 `selec * from users;`，解析器会抛出语法错误，因为 `selec` 拼写错误，不是有效的 SQL 关键字。

### 解释 (Explanation)
- **词法分析**：将 SQL 字符串拆分为独立的词法标记（识别出关键字，如 `select`、`from`，以及表名、字段名等）。
- **语法分析**：构建抽象语法树（AST），并根据规则验证你输入的 SQL 语句是否符合 MySQL 语法。
- *注意*：解析器**不会**检查表或字段是否存在，它只管语法对不对。

---

## 5. 第四步（1）：预处理器

### 直觉 (Intuition)
预处理器是事实核查员。现在句子语法是对的了，它要检查你提到的东西在真实的数据库世界里是否真的存在。

### 例子 (Example)
查询：`SELECT * FROM non_existent_table;`
解析器会让它通过（因为语法完全正确），但预处理器会报错：`Table 'non_existent_table' doesn't exist`。

### 解释 (Explanation)
- 检查 SQL 语句中涉及的表和字段是否真实存在。
- 将 `SELECT *` 中的 `*` 符号扩展为该表上的所有实际列。

---

## 6. 第四步（2）：优化器

### 直觉 (Intuition)
优化器就像 GPS 导航系统。去目的地（获取数据）可能有好几条不同的路线，GPS 会评估路况、距离和收费，为你选择一条最快、最省油的路线。

### 例子 (Example)
假设执行 `SELECT id FROM product WHERE id > 1 AND name LIKE 'i%';`（`id` 是主键，`name` 是二级普通索引）。
优化器意识到扫描 `name` 二级索引更划算，因为这是一个“覆盖索引”（二级索引叶子节点存有主键 `id`），直接就能拿到所需数据，避免了去查主键索引的巨大成本。

### 解释 (Explanation)
- 负责确定 SQL 查询语句的**执行计划**。
- 基于查询成本（CPU和I/O开销）的考虑，在有多个索引可用时决定选择使用哪个索引，或者决定是否使用全表扫描。
- 可以通过在查询语句最前面加上 `EXPLAIN` 命令来查看优化器选择的执行计划。

---

## 7. 第四步（3）：执行器

### 直觉 (Intuition)
执行器是实际按照 GPS 路线开车的司机。它一步一步跟底层引擎（存储引擎）沟通，发号施令：“给我第一条符合条件的数据”，检查一下，然后再说：“给我下一条”，直到查完为止。

### 例子 (Example)
执行 `SELECT * FROM product WHERE id = 1;`（主键查询）：
执行器告诉存储引擎：“找到 id=1 的第一条记录”。引擎通过 B+ 树找到后返回。执行器检查符合条件，发给客户端。执行器再要下一条，引擎说“没了”，循环退出。

### 解释 (Explanation)
- 根据优化器生成的执行计划，通过调用存储引擎提供的 API（如 `read_first_record`, `read_record`）来真正执行 SQL 语句。
- 常见的执行流程：
  1. **主键索引查询**：访问类型为 `const`，通常只调用一次引擎接口就能找到数据。
  2. **全表扫描**：访问类型为 `ALL`，执行器会循环调用接口让引擎逐行读取，Sever 层自己一条条判断是否符合条件。

### 7.1 进阶概念：索引下推 (Index Condition Pushdown, ICP)

#### 直觉 (Intuition)
以前是仓库工人（存储引擎）把所有可能相关的箱子（数据行）全搬给经理（Server 层），让经理一个个检查里面是不是我们要的货。现在有了“下推”，经理把具体的筛选条件直接交给了工人，工人在仓库里直接过滤掉不合格的箱子，只把完全符合条件的搬过去，省去了无数趟白跑。

#### 例子 (Example)
假设有联合索引 `(age, reward)`，执行查询：`age > 20 AND reward = 100000`。
- *没有索引下推*：引擎根据 `age > 20` 在二级索引找到记录后，获取主键，进行**回表**操作查出完整数据发给 Server 层。Server 层再判断 `reward` 是否等于 100000。
- *使用索引下推*：引擎在二级索引找到 `age > 20` 后，**先不回表**，直接在当前索引数据里判断 `reward = 100000` 是否成立。不成立直接跳过；成立才去执行**回表**获取完整记录返回给 Server 层。

#### 解释 (Explanation)
- 减少了二级索引查询时的**回表操作**（回表：根据二级索引叶子节点的主键值去主键索引 B+ 树获取完整行数据）。
- 它将 Server 层负责的一部分判断工作交给了存储引擎层去处理。
- 当执行计划的 `Extra` 字段显示 `Using index condition` 时，说明触发了索引下推优化，极大提高了查询效率。
