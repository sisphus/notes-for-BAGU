# 自测题：MySQL `SELECT` 语句执行流程

## 1. MySQL 整体架构
**问题：** MySQL 架构的两个主要分层是什么？它们各自的主要职责是什么？
<details>
<summary>答案</summary>

- **Server 层：** 负责建立连接、分析和执行 SQL。大部分核心功能（如连接器、查询缓存、解析器、预处理器、优化器、执行器等）都在这里实现。所有的内置函数以及跨存储引擎的功能也都属于这一层。
- **存储引擎层：** 负责数据的存储和提取。采用插件式架构，支持 InnoDB、MyISAM 等引擎。它负责管理实际的文件数据以及索引结构（如 InnoDB 默认的 B+ 树）。
</details>

## 2. 连接器
**问题：** 如果管理员在一个用户已经建立连接的情况下修改了该用户的权限，新的权限会影响该用户当前的会话吗？为什么？
<details>
<summary>答案</summary>

不会。当连接建立时，连接器会读取并保存该用户的权限。该特定连接中的所有后续操作都依赖于最开始缓存的这些权限。修改后的新权限只对修改完成后新创建的连接生效。
</details>

**问题：** 有哪两种方法可以解决 MySQL 中由于使用长连接导致内存占用过多的问题？
<details>
<summary>答案</summary>

1. **定期断开长连接：** 在连接使用一段时间后，或者程序执行过一个占用大量内存的大型查询后，主动断开连接，下次查询时再重新建立。
2. **客户端主动重置连接：** 在 MySQL 5.7 及以上版本中，可以在大型操作后调用 `mysql_reset_connection()` 函数接口来重置连接状态。这会将连接恢复到刚刚创建完时的状态，释放内存资源，且无需重新进行 TCP 握手和权限验证。
</details>

## 3. 查询缓存
**问题：** 为什么在 MySQL 8.0 版本中彻底删除了查询缓存（Query Cache）功能？
<details>
<summary>答案</summary>

因为查询缓存非常“鸡肋”，对于更新频繁的表来说效率极低。只要一个表发生任何更新操作，该表相关的所有查询缓存都会被清空。这意味着缓存经常在发挥作用之前就被清空了（相当于白缓存了），不但没提升性能，反而增加了系统的开销。
</details>

## 4. 解析器与预处理器
**问题：** 如果你查询了一张不存在的表，MySQL 会在执行流程的哪个具体阶段报错？
<details>
<summary>答案</summary>

会在**预处理器**阶段报错。
**解析器**只负责词法和语法分析（例如检查你的 SQL 语句拼写和结构是否符合语法规则）。真正检查数据库中“表或字段是否存在”的工作，是由预处理器来完成的。
</details>

## 5. 优化器
**问题：** 优化器的主要作用是什么？你如何查看它最终选择了哪种方案？
<details>
<summary>答案</summary>

优化器的主要作用是为 SQL 查询语句确定最佳的**执行计划**。基于查询成本（CPU和I/O）的考虑，当表中有多个索引时，它会决定选择使用哪一个索引，或者判断是否需要全表扫描。
你可以在 SQL 查询语句最前面加上 `EXPLAIN` 命令，以此来输出并查看该语句的执行计划（例如通过 `key` 字段查看使用了哪个索引）。
</details>

## 6. 执行器与索引下推
**问题：** 请描述索引下推（Index Condition Pushdown, ICP）是如何提升查询效率的。
<details>
<summary>答案</summary>

- **没有索引下推时**：存储引擎根据二级索引定位到部分符合条件的记录后，必须进行**回表**操作（根据二级索引找到的主键值去查主键 B+ 树），取出完整的行数据返回给 Server 层。然后由 Server 层去判断剩余的查询条件是否符合，不符合则丢弃。
- **使用索引下推后**：Server 层将那些包含在联合索引中但无法用来做范围匹配的条件交给了存储引擎。存储引擎在二级索引中直接判断这些条件是否成立，如果不成立直接跳过，**只有条件成立时才去执行回表操作**。
这极大地减少了昂贵的回表次数，提高了查询效率。
</details>
