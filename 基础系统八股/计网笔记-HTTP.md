

# “2.1 TCP/IP 网络模型有哪几层？”

## ✅ TCP/IP 网络模型总览（4 层）

TCP/IP 是互联网的基础通信模型，由 **4 层**组成，每层都有不同的职责：

| 层级                           | 主要作用                                 | 常见协议/设备        | 关键词记忆         |
| ------------------------------ | ---------------------------------------- | -------------------- | ------------------ |
| **应用层**                     | 为用户提供各种网络服务（例如网页、邮件） | HTTP、FTP、DNS、SMTP | “用户能看到的功能” |
| **传输层**                     | 在两台主机的应用之间提供端到端通信       | TCP、UDP             | “端到端、可靠传输” |
| **网络层**                     | 负责数据包在不同网络间的路由与转发       | IP、ICMP、ARP        | “跨网找路”         |
| **网络接口层（或数据链路层）** | 处理数据在物理链路上传输                 | Ethernet、Wi-Fi、MAC | “在网线上传”       |

------

## 🔹 1. 应用层（Application Layer）

- 提供具体的网络服务功能，例如：
  - HTTP：网页访问
  - FTP：文件传输
  - DNS：域名解析
  - SMTP：邮件发送
- 不关心底层数据传输，只负责生成和解析应用数据。

📌 **口诀：**“只管内容，不管运输。”

------

## 🔹 2. 传输层（Transport Layer）

- 建立**端到端通信**，保证数据可靠或高效传输。
- 两大协议：
  - **TCP**：面向连接，可靠（如网页访问）
  - **UDP**：无连接，快但不可靠（如视频通话）
- 提供端口号区分不同应用。

📌 **口诀：**“谁和谁在通信。”

------

## 🔹 3. 网络层（Internet Layer）

- 负责**寻址和路由**，让数据能跨网络传输。
- 核心协议：**IP**（Internet Protocol）
  - IPv4/IPv6 地址
  - 路由选择（根据 IP 找目的地）
- 辅助协议：ICMP、ARP 等。

📌 **口诀：**“帮你找到对方的家。”

------

## 🔹 4. 网络接口层（Link/Network Access Layer）

- 负责**物理传输**，如网线、Wi-Fi。
- 把 IP 包封装成帧，在局域网中传递。
- 使用 MAC 地址识别设备。

📌 **口诀：**“数据上网的最后一步。”

------

## 🧠 记忆技巧

> “应传网链” → 从上到下（用户 → 内核 → 硬件）

| 层次关系   | 对应 OSI 七层模型    |
| ---------- | -------------------- |
| 应用层     | 应用 + 表示 + 会话层 |
| 传输层     | 传输层               |
| 网络层     | 网络层               |
| 网络接口层 | 数据链路 + 物理层    |

# “2.2 键入网址到网页显示，期间发生了什么？”

## 🚀 整体流程一句话概括：

> **输入网址 → DNS 解析 → 建立 TCP 连接 → 发送 HTTP 请求 → 服务器响应 → 浏览器渲染 → 断开连接**

------

## 🔹 步骤详解（7 步复习法）

| 步骤                             | 发生的事                                                    | 涉及层          | 关键点                                                    |
| -------------------------------- | ----------------------------------------------------------- | --------------- | --------------------------------------------------------- |
| **1. 输入网址**                  | 用户在浏览器输入 URL（如 www.example.com）                  | 应用层          | 浏览器先检查缓存（DNS 缓存、HTTP 缓存）                   |
| **2. DNS 解析**                  | 把域名解析成 IP 地址                                        | 应用层 → 网络层 | 查询顺序：浏览器缓存 → 系统缓存 → 本地 hosts → DNS 服务器 |
| **3. 建立 TCP 连接（三次握手）** | 浏览器与服务器通过 TCP 建立可靠连接                         | 传输层          | SYN → SYN+ACK → ACK                                       |
| **4. 发送 HTTP 请求**            | 浏览器向服务器发送请求报文                                  | 应用层          | 请求行、请求头、请求体（GET/POST 等）                     |
| **5. 服务器处理并响应**          | 服务器返回响应报文（HTML、CSS、JS）                         | 应用层          | 含状态码（200、404 等）、响应头、正文                     |
| **6. 浏览器渲染页面**            | 解析 HTML → 下载资源 → 构建 DOM + CSSOM → 渲染树 → 绘制页面 | 应用层          | 执行 JS、布局、绘制                                       |
| **7. 断开连接（四次挥手）**      | 数据传输完毕，断开 TCP 连接                                 | 传输层          | FIN → ACK → FIN → ACK                                     |

## 🧩 数据在各层的传递过程

**封装方向（发送时）：**

> 应用层（HTTP） → 传输层（TCP） → 网络层（IP） → 链路层（MAC帧）

**解封方向（接收时）：**

> 链路层 → 网络层 → 传输层 → 应用层

📌 每层都加上自己的“首部(header)”来标识和处理数据。

------

## 🌐 一图记忆：

```
输入网址
   ↓
DNS 查询 IP
   ↓
TCP 三次握手
   ↓
HTTP 请求/响应
   ↓
浏览器渲染页面
   ↓
TCP 四次挥手
```

# “2.3 Linux 系统是如何收发网络包的？”

## 🧭 总览一句话

> **应用层发数据 → 内核协议栈封装 → 网卡驱动发送 → 通过物理介质传输 → 对方网卡接收 → 协议栈解封装 → 应用层读取。**

------

## 🚀 一、发送数据流程（发送方向）

| 步骤                         | 发生的事                                                  | 层级            | 关键词                  |
| ---------------------------- | --------------------------------------------------------- | --------------- | ----------------------- |
| **1. 应用层调用 `send()`**   | 应用程序（如浏览器）调用 `send()` 把数据交给内核          | 应用层 → 传输层 | 数据写入 Socket 缓冲区  |
| **2. 传输层封装 TCP/UDP 头** | 加上源端口、目的端口、校验和等                            | 传输层          | TCP 分段 / UDP 封包     |
| **3. 网络层封装 IP 头**      | 添加源 IP、目的 IP                                        | 网络层          | 选路、TTL、分片         |
| **4. 数据链路层封装帧头**    | 加上源 MAC、目的 MAC、帧校验 FCS                          | 链路层          | 查询 ARP 表得到对方 MAC |
| **5. 网卡驱动发送**          | 网卡通过 DMA 从内存取数据 → 发到物理网络（电信号/光信号） | 硬件层          | DMA + 中断机制          |
| **6. 数据离开主机**          | 经交换机/路由器 → 传输到目标主机                          | 物理层          | 实际传输媒介            |

📌 **口诀：**“应用写 → 协议封 → 网卡发”。

## 🛰️ 二、接收数据流程（接收方向）

| 步骤                              | 发生的事                            | 层级       | 关键词             |
| --------------------------------- | ----------------------------------- | ---------- | ------------------ |
| **1. 网卡收到信号**               | 将电信号转成比特流放入 Ring Buffer  | 硬件层     | DMA 把数据写入内存 |
| **2. 触发中断通知内核**           | 内核调用网卡驱动 → 数据进入内核空间 | 驱动层     | 中断/轮询 NAPI     |
| **3. 协议栈层层解包**             | 链路层 → 网络层 → 传输层            | 内核协议栈 | 检查校验、重组分片 |
| **4. 数据进入 Socket 接收缓冲区** | TCP 按序重组、确认 ACK              | 传输层     | 滑动窗口           |
| **5. 应用层调用 `recv()`**        | 应用程序读取数据并处理              | 应用层     | 用户态读取         |

📌 **口诀：**“网卡收 → 内核拆 → 应用读”。

------

## ⚙️ 三、关键内核机制要记住

| 名称                            | 作用                   | 说明                         |
| ------------------------------- | ---------------------- | ---------------------------- |
| **Socket 缓冲区**               | 应用与内核通信的桥梁   | send/recv 读写此缓冲区       |
| **DMA（Direct Memory Access）** | 网卡直接与内存交换数据 | 减少 CPU 拷贝                |
| **中断 & NAPI**                 | 通知内核有新包到达     | NAPI 混合中断 + 轮询提高性能 |
| **协议栈**                      | 实现 TCP/IP 各层逻辑   | Linux 内核中的核心模块       |

------

## 📊 一图速记

```
发送：App → Socket缓冲区 → TCP/IP协议栈封装 → 网卡 → 物理网络
接收：网卡 → 协议栈解封装 → Socket缓冲区 → App读取
```

# **“3.1 HTTP 常见面试题”** 

## 🧭 一、HTTP 基础概念

| 内容            | 速记                                                         |
| --------------- | ------------------------------------------------------------ |
| **HTTP 是什么** | 超文本传输协议（HyperText Transfer Protocol），用于客户端与服务器通信。 |
| **特点**        | 基于请求-响应模型、无状态（每次请求独立）、基于 TCP。        |
| **默认端口**    | 80（HTTP），443（HTTPS）。                                   |

📌 **一句话记忆：**“HTTP 负责内容交流，TCP 负责可靠传输。”

## 🚀 二、HTTP 请求与响应结构

### 请求报文

```
请求行：GET /index.html HTTP/1.1
请求头：Host, User-Agent, Cookie, Content-Type ...
请求体：POST 提交的数据
```

### 响应报文

```
状态行：HTTP/1.1 200 OK
响应头：Content-Type, Set-Cookie, Cache-Control ...
响应体：HTML/JSON 文件内容
```

## ⚙️ 三、常见面试题速背表

| 问题                               | 答案要点                                                     |
| ---------------------------------- | ------------------------------------------------------------ |
| **1. HTTP 常见方法？**             | GET（取资源）、POST（提交）、PUT（更新）、DELETE（删除）、HEAD、OPTIONS。 |
| **2. GET 和 POST 区别？**          | GET 参数在 URL 中、幂等、可缓存；POST 在 body 中、非幂等。   |
| **3. 状态码含义？**                | 1xx 信息、2xx 成功、3xx 重定向、4xx 客户端错误、5xx 服务器错误。 例：200 OK、301 永久重定向、404 Not Found、500 内部错误。 |
| **4. Cookie 与 Session 区别？**    | Cookie 存客户端；Session 存服务器；SessionID 通过 Cookie 传递。 |
| **5. HTTP 是无状态的是什么意思？** | 服务器不会记住前一次请求，需要 Cookie/Session 实现“有状态”。 |
| **6. HTTP 与 HTTPS 区别？**        | HTTPS = HTTP + SSL/TLS 加密；更安全（防窃听、篡改、伪造）。  |
| **7. HTTP/1.0 vs HTTP/1.1？**      | 1.1 支持长连接（Keep-Alive）、管线化（Pipelining）、分块传输。 |
| **8. HTTP/2 优点？**               | 多路复用、头部压缩（HPACK）、二进制分帧、服务器推送。        |
| **9. HTTP/3 特点？**               | 基于 QUIC 协议（UDP），减少握手延迟，解决队头阻塞。          |
| **10. 缓存机制？**                 | 强缓存：Expires / Cache-Control；协商缓存：ETag / Last-Modified。 |

## 📊 四、三代 HTTP 对比速览

| 协议         | 连接特性             | 性能优化     | 底层协议    |
| ------------ | -------------------- | ------------ | ----------- |
| **HTTP/1.1** | 长连接（Keep-Alive） | 但有队头阻塞 | TCP         |
| **HTTP/2**   | 多路复用             | 压缩+推送    | TCP         |
| **HTTP/3**   | 无队头阻塞           | 更快握手     | UDP（QUIC） |

📌 **口诀：**

> 1.0：一请求一连接
>  1.1：长连接但排队
>  2：多路并行
>  3：UDP 飞快传

------

## 💡 小结一句话：

> HTTP 从 **简单文本协议** → **高性能多路传输协议**，
>  不断进化以**提升速度、安全性与用户体验**。

# 3.2 HTTP/1.1 如何优化？

## 🚀 一、问题背景

HTTP/1.0 最大缺陷：

- **每次请求都要新建 TCP 连接**（三次握手 + 四次挥手 = 慢）
- 无法复用连接 → 延迟高、浪费带宽

📌 **一句话：**“建连接太频繁，浪费又慢。”

## ⚙️ 二、HTTP/1.1 的主要优化手段

| 优化点                                           | 说明                                      | 关键机制                                   |
| ------------------------------------------------ | ----------------------------------------- | ------------------------------------------ |
| **1. 长连接（Persistent Connection）**           | 一个 TCP 连接可复用多次请求               | 头部字段：`Connection: keep-alive`         |
| **2. 管线化（Pipelining）**                      | 多个请求可连续发送，不必等上一个响应      | 提高并发度，但仍可能受“队头阻塞”影响       |
| **3. 分块传输编码（Chunked Transfer-Encoding）** | 不知道内容总长度时，边生成边传输          | `Transfer-Encoding: chunked`               |
| **4. 增加缓存控制机制**                          | 强缓存（Cache-Control）、协商缓存（ETag） | 降低重复请求、减少带宽消耗                 |
| **5. 增加 Host 字段**                            | 支持虚拟主机（一个 IP 多个域名）          | 头部：`Host: www.example.com`              |
| **6. 状态码更丰富**                              | 206（部分内容）、100（继续）等            | 支持断点续传、分块请求                     |
| **7. 内容协商机制**                              | 根据语言/格式自动选择资源                 | 头部：`Accept-Language`、`Accept-Encoding` |
| **8. 支持范围请求（Range Request）**             | 只请求文件的部分内容                      | 头部：`Range: bytes=0-1024`（断点续传）    |

## 📊 三、请求复用机制示意

```
HTTP/1.0: 每个请求 = 新建 TCP 连接 ❌
HTTP/1.1: 一个 TCP 连接可多次复用 ✅
```

例：

```
Client ----TCP连接----> Server
GET /a.jpg
GET /b.jpg
GET /c.jpg
```

所有请求共用一个连接，显著减少延迟。

------

## 🧩 四、局限性（为 HTTP/2 铺路）

| 问题                                  | 说明                                         |
| ------------------------------------- | -------------------------------------------- |
| **队头阻塞（Head-of-line Blocking）** | 管线化请求仍按顺序返回，前一个慢则后面都卡   |
| **Header 重复冗长**                   | 每个请求头都带 Cookie / User-Agent，浪费带宽 |
| **TCP 连接共享不足**                  | 虽可复用，但浏览器通常限制同域 6 个连接      |

📌 **总结：**HTTP/1.1 改善连接效率，但仍未彻底解决性能瓶颈。

# 3.3 HTTPS：RSA 握手解析

## 🌐 一、HTTPS = HTTP + SSL/TLS

| 项目     | HTTP   | HTTPS                      |
| -------- | ------ | -------------------------- |
| 端口     | 80     | 443                        |
| 传输协议 | 明文   | 加密传输（SSL/TLS）        |
| 安全性   | 不安全 | 保密性 + 完整性 + 身份验证 |
| 关键目的 | 无     | 防窃听、防篡改、防伪造     |

📌 **一句话记忆：**

> HTTPS 让 HTTP 的内容在“加密隧道”里传输。

## 🔐 二、RSA 握手核心思路

> “用非对称加密建立对称加密密钥，再用对称加密传输数据。”

## ⚙️ 三、握手流程（以 TLS 1.2 + RSA 为例）

| 步骤                            | 名称                                                 | 说明                                                    | 加密方式 |
| ------------------------------- | ---------------------------------------------------- | ------------------------------------------------------- | -------- |
| **① ClientHello**               | 客户端发起握手                                       | 告诉服务器支持的 TLS 版本、加密算法、随机数 **Random1** | 无加密   |
| **② ServerHello + Certificate** | 服务器响应                                           | 发送服务器随机数 **Random2** + 数字证书（含公钥）       | 无加密   |
| **③ 客户端验证证书**            | 验证服务器身份                                       | 校验证书签名是否合法（CA 签发）                         | 无加密   |
| **④ 生成 Pre-Master Secret**    | 客户端生成随机密钥 **Pre-Master**                    | 用服务器公钥（RSA）加密后发送                           |          |
| **⑤ 服务器解密 Pre-Master**     | 服务器用私钥解密得到 Pre-Master                      | 非对称加密解密                                          |          |
| **⑥ 双方计算对称密钥**          | 根据 Random1 + Random2 + Pre-Master 生成 Session Key | 对称密钥生成                                            |          |
| **⑦ 握手结束，开始加密通信**    | 后续 HTTP 数据用 Session Key 加密                    | 对称加密（AES等）                                       |          |

## 🧩 四、可视化图解（简化）

```
Client                   Server
  | ---- ClientHello ----> |
  | <--- ServerHello + Cert |
  | ---- Pre-Master (RSA加密) ----> |
  | <=== 对称密钥已建立 ===> |
  | ---- Encrypted HTTP ----> |
```

------

## 🔏 五、安全机制说明

| 安全目标     | 实现方式                    |
| ------------ | --------------------------- |
| **机密性**   | 对称加密（AES）             |
| **身份验证** | 服务器证书（CA签名）        |
| **完整性**   | 消息摘要（MAC / HMAC）      |
| **密钥交换** | RSA 公钥加密传输 Pre-Master |

📌 RSA 只用于**密钥交换阶段**，不是用于整个通信加密。

##  六、RSA 握手的缺陷（为 ECDHE 铺路）

| 缺点                       | 说明                              |
| -------------------------- | --------------------------------- |
| **无前向安全性**           | 服务器私钥泄露 → 历史通信全被解密 |
| **计算开销大**             | 公钥加密解密性能低                |
| **易被中间人伪造证书攻击** | 若根证书链不安全，则被信任伪造    |

👉 因此，现代 HTTPS 多用 **ECDHE 握手**（椭圆曲线 Diffie-Hellman）取代 RSA。

# 3.4 HTTPS：ECDHE 握手解析

## 🌐 一、为什么要用 ECDHE？

RSA 握手的缺点：

- ❌ 无前向安全性（私钥泄露 → 历史通信全被破解）
- ❌ 计算量大（公钥加密解密开销高）

📌 **解决方案：**
 👉 使用 **ECDHE（Elliptic Curve Diffie-Hellman Ephemeral）** 动态生成密钥。
 它提供 **“前向安全性（Forward Secrecy）”**。

## ⚙️ 二、ECDHE 握手核心思想

> “客户端和服务器各自生成临时密钥对 → 交换公钥 → 双方计算出相同的对称密钥（Session Key）。”

📌 特点：

- 不再用 RSA 加密 Pre-Master，而是用椭圆曲线 Diffie-Hellman 计算。
- 每次握手生成不同密钥 → 过去通信无法被解密。

## 🔐 三、TLS 1.2 下 ECDHE 握手流程（简化版）

| 步骤                                                | 名称                                             | 内容                       | 加密方式 |
| --------------------------------------------------- | ------------------------------------------------ | -------------------------- | -------- |
| **① ClientHello**                                   | 客户端发送支持的加密套件 + 随机数 Random1        | 明文                       |          |
| **② ServerHello + Certificate + ServerKeyExchange** | 服务端返回 Random2、证书、公钥（含椭圆曲线参数） | 明文（证书签名保证完整性） |          |
| **③ ClientKeyExchange**                             | 客户端生成临时公私钥对，发送公钥                 | 明文（无法推算出私钥）     |          |
| **④ 双方计算共享密钥**                              | 各自用对方公钥 + 自己私钥 计算相同 Session Key   | 数学计算（ECDH）           |          |
| **⑤ Finished 阶段**                                 | 双方验证握手数据完整性                           | 对称加密（Session Key）    |          |
| **⑥ 开始加密通信**                                  | HTTP 数据用 Session Key 加密传输                 | AES/GCM 等对称算法         |          |

## 🔑 四、图解流程（超简化）

```
Client                Server
  | ---- ClientHello ----> |
  | <--- ServerHello + Cert + ECDHE参数 |
  | ---- Client ECDHE公钥 ----> |
  | <=== 双方计算Session Key ===> |
  | ---- Finished(加密验证) ----> |
  | <--- Finished(加密验证) ---- |
  | <=== 开始加密HTTP通信 ===> |
```

------

## 🧠 五、ECDHE 优势总结

| 优势                                | 说明                               |
| ----------------------------------- | ---------------------------------- |
| ✅ **前向安全性（Forward Secrecy）** | 即使服务器私钥泄露，历史会话仍安全 |
| ✅ **更高性能**                      | 椭圆曲线算法比 RSA 更快            |
| ✅ **动态密钥**                      | 每次握手生成新的会话密钥           |
| ✅ **更安全的身份认证**              | 仍通过数字证书验证服务器身份       |

📌 **一句话记忆：**

> “ECDHE = 临时密钥交换 + 椭圆曲线加速 + 前向安全。”

------

## ⚙️ 六、TLS 1.3 的进一步优化

| 项目     | TLS 1.2           | TLS 1.3           |
| -------- | ----------------- | ----------------- |
| 握手轮次 | 2-RTT（两次往返） | 1-RTT（一次往返） |
| 密钥交换 | ECDHE 可选        | ECDHE 强制        |
| 证书验证 | 分阶段            | 合并发送          |
| 加密算法 | 可选多种          | 精简高效（AEAD）  |

📌 TLS 1.3 默认使用 **ECDHE + AEAD 加密**，
 并支持 **0-RTT** 快速恢复连接（Session Resumption）。

## 🧩 七、对比 RSA 与 ECDHE 握手

| 项目     | RSA 握手                | ECDHE 握手             |
| -------- | ----------------------- | ---------------------- |
| 密钥生成 | 由客户端生成 Pre-Master | 双方协商临时密钥       |
| 安全性   | 无前向安全              | 具备前向安全           |
| 加密方式 | 非对称加密传密钥        | 数学协商共享密钥       |
| 性能     | 慢（RSA 运算多）        | 快（椭圆曲线轻量）     |
| 实际应用 | 旧版 HTTPS              | 现代主流（TLS1.2/1.3） |

# 3.5 HTTPS 如何优化？

## 🌐 一、为什么要优化 HTTPS？

HTTPS 虽安全，但存在性能问题：

- 多次握手（TLS + TCP）增加延迟
- 加解密计算耗时
- 证书验证与管理复杂

📌 **一句话：**“安全带来的代价是握手慢、计算多。”



## 🚀 二、优化方向总览

| 优化方向           | 核心手段                                    |
| ------------------ | ------------------------------------------- |
| 🔹 减少握手延迟     | TLS 1.3、Session Resumption、0-RTT          |
| 🔹 降低加解密成本   | 使用硬件加速、轻量算法（AES-GCM、ChaCha20） |
| 🔹 减少证书验证开销 | 启用 OCSP Stapling、证书缓存                |
| 🔹 减少连接次数     | HTTP/2 多路复用、连接复用                   |
| 🔹 加快域名解析     | DNS 缓存、DNS over HTTPS                    |

## ⚙️ 三、TLS 优化手段详解

### ① **TLS 1.3：握手轮次减少**

- 从 **2-RTT → 1-RTT**
- 合并 “密钥交换 + 证书验证”
- 默认使用 **ECDHE**（更快、更安全）

📌 **结果：** 握手延迟降低约 50%。

------

### ② **会话复用（Session Resumption）**

| 方式                       | 原理                                    | 优点         |
| -------------------------- | --------------------------------------- | ------------ |
| **Session ID**             | 服务器保存旧会话 ID，下次直接恢复       | 减少重握手   |
| **Session Ticket**         | 客户端保存加密 Ticket，服务端无状态恢复 | 减少内存占用 |
| **Session Resumption PSK** | TLS1.3 新版机制，预共享密钥恢复会话     | 安全+高效    |

📌 **效果：** 省去完整握手过程。

### ③ **0-RTT（Zero Round Trip Time）**

- **TLS 1.3 新特性**
- 客户端在恢复会话时**不等握手完成**就能发第一个请求
- 适合短连接或频繁访问的场景

⚠️ 缺点：可能被重放攻击（仅限幂等请求）。

📌 **一句话记忆：**“之前聊过的，下次直接说话。”

------

### ④ **OCSP Stapling**

- 证书吊销验证优化
- 由**服务器提前缓存 CA 签名的状态**，直接发给客户端
- 避免客户端再请求 CA，提高加载速度。

------

### ⑤ **硬件与算法优化**

| 优化                                | 说明              |
| ----------------------------------- | ----------------- |
| ✅ 硬件加密加速（AES-NI 指令集）     | 减少 CPU 加密负担 |
| ✅ 使用轻量算法（ChaCha20-Poly1305） | 移动端性能好      |
| ✅ 启用会话缓存/复用                 | 避免重复计算      |

------

## 🌐 四、HTTP 层的配合优化

| 手段                   | 原理                | 效果         |
| ---------------------- | ------------------- | ------------ |
| **HTTP/2 多路复用**    | 复用一条 TLS 连接   | 减少握手数量 |
| **CDN 部署**           | 就近接入            | 降低 RTT     |
| **HSTS（强制 HTTPS）** | 避免多次 301 重定向 | 加快首访速度 |
| **DNS 缓存**           | 减少 DNS 查询延迟   | 提前解析域名 |

## 📊 五、可视化流程对比

```
传统 HTTPS (TLS1.2)
Client ----握手1----> Server
Client ----握手2----> Server
Client ----HTTP请求----> Server

优化后 (TLS1.3 + 0-RTT)
Client ----握手同时发送HTTP请求----> Server
```

👉 少 1 次往返延迟，整体快约 30~50%。

# 3.6 HTTP/2 牛逼在哪？

## 🚀 一、HTTP/2 为什么出现？

HTTP/1.1 的三大瓶颈：

1. **队头阻塞（HOL Blocking）**：一个请求慢 → 后面全被堵。
2. **头部冗长**：重复发送 Cookie、User-Agent 等。
3. **多连接浪费**：浏览器每域最多开 6 条 TCP。

📌 **一句话：**“连接多、头太大、排队慢。”

## ⚙️ 二、HTTP/2 的四大核心特性

| 特性                               | 说明                                          | 优势                     |
| ---------------------------------- | --------------------------------------------- | ------------------------ |
| **1️⃣ 二进制分帧（Binary Framing）** | 请求与响应不再是纯文本，而是二进制帧（Frame） | 高效、易解析、减少体积   |
| **2️⃣ 多路复用（Multiplexing）**     | 同一 TCP 连接上并行发送多个流（Stream）       | 消除队头阻塞、减少连接数 |
| **3️⃣ 头部压缩（HPACK）**            | 只传变化部分 + 静态表索引                     | 减少带宽消耗             |
| **4️⃣ 服务器推送（Server Push）**    | 服务器可主动推送资源                          | 减少等待时间             |

📌 **口诀：**“二进制、多路跑、头压缩、服务器先送包。”

## 🧩 三、工作机制（简化示意）

```
HTTP/1.1:  
Request1 ---Response1  
Request2 ---Response2  ← 顺序阻塞  

HTTP/2:  
Stream1: Req/Res交错  
Stream2: Req/Res交错  
Stream3: Req/Res交错  
→ 同一连接并行！
```

------

## ⚙️ 四、性能提升总结

| 优化点     | HTTP/1.1             | HTTP/2             |
| ---------- | -------------------- | ------------------ |
| 连接复用   | ❌ 每次新建或队列排队 | ✅ 一条连接多流并发 |
| 数据格式   | 文本                 | 二进制帧           |
| 头部传输   | 重复                 | HPACK 压缩         |
| 请求顺序   | 阻塞                 | 并行               |
| 服务端能力 | 被动响应             | 可主动推送资源     |

📌 实测可减少 **页面加载延迟 30%~60%**。

## 🔐 五、兼容性 & 限制

- 向下兼容 HTTP/1.1（通过 TLS ALPN 协商）
- 依然基于 **TCP** → 仍可能存在**TCP 队头阻塞**（→ HTTP/3 解决）

# 3.7 HTTP/3：强势来袭

## 🚀 一、HTTP/3 是谁？

> **HTTP/3 = HTTP/2 + QUIC（基于 UDP）**

📌 **一句话：**

> “HTTP/3 把 HTTP 从 TCP 上搬到了 UDP 上。”

## ⚙️ 二、为什么要抛弃 TCP？

| 问题             | HTTP/1.1 & 2          |
| ---------------- | --------------------- |
| **TCP 队头阻塞** | 一个包丢，全连接卡住  |
| **握手慢**       | TCP + TLS 多次往返    |
| **迁移困难**     | IP 或端口变动就得重连 |

📌 HTTP/2 虽有多路复用，但**受限于 TCP**，多流共享一条管道仍会被堵。

## 🔑 三、HTTP/3 的核心：**QUIC 协议**

| 特性                  | 说明                         | 优势                         |
| --------------------- | ---------------------------- | ---------------------------- |
| **基于 UDP**          | 自实现可靠传输（不依赖 TCP） | 无队头阻塞                   |
| **内置 TLS 1.3 加密** | 握手一次完成安全协商         | 1-RTT / 0-RTT                |
| **多路复用**          | 每个 Stream 独立传输         | 一个丢包不影响其他流         |
| **连接迁移**          | 连接 ID 不依赖 IP            | 网络切换不中断（Wi-Fi ⇋ 4G） |
| **更快恢复**          | 丢包重传粒度更小             | RTT 低、延迟少               |

📌 **一句话记忆：**

> QUIC = “UDP 版 TCP + TLS + 多路复用”。

## 🧩 四、握手流程（TLS1.3 + QUIC）

```
HTTP/2 (TCP)
Client --TCP握手---> Server
Client --TLS握手---> Server
(共2-RTT)

HTTP/3 (QUIC)
Client --QUIC握手 + TLS1.3--> Server
(仅1-RTT，甚至0-RTT)
```

------

## 📊 五、性能对比总结

| 对比项   | HTTP/2（TCP）      | HTTP/3（UDP+QUIC）    |
| -------- | ------------------ | --------------------- |
| 传输层   | TCP                | UDP                   |
| 握手耗时 | 2RTT               | 1RTT / 0RTT           |
| 队头阻塞 | 有                 | 无                    |
| 多路复用 | 受 TCP 限制        | 完全独立              |
| 连接迁移 | 不支持             | 支持（Connection ID） |
| 安全协议 | 外部 TLS           | 内置 TLS1.3           |
| 应用场景 | 传统浏览器、服务端 | 移动端、CDN、实时通信 |

📌 实测：HTTP/3 比 HTTP/2 平均快 30%~40%。

## ✅ 七、总结一句话

> **HTTP/3 是互联网的“快车道”版本：**
>  它彻底解决了 TCP 队头阻塞、慢握手、断线重连慢的问题，
>  让网页加载、视频流、实时通信都更流畅。

# **“3.8 既然有 HTTP 协议，为什么还要有 RPC？”** 

## 🌐 一、HTTP vs RPC：都是“远程通信”，但定位不同

| 对比项     | **HTTP**                 | **RPC（Remote Procedure Call）** |
| ---------- | ------------------------ | -------------------------------- |
| 核心目标   | 资源传输（文档/接口）    | 调用远程函数（方法级通信）       |
| 通信模型   | 基于请求-响应（REST）    | 基于函数调用（方法名 + 参数）    |
| 协议层     | 应用层协议（HTTP/1/2/3） | 可基于 TCP / HTTP / QUIC         |
| 序列化方式 | 文本（JSON、XML）        | 高效二进制（Protobuf、Thrift）   |
| 调用粒度   | 面向资源（URI）          | 面向服务（函数 / 方法）          |
| 性能       | 较高延迟、通用性强       | 更低延迟、性能优化更深           |
| 应用场景   | Web 服务、API 接口       | 微服务内部通信、高性能后端       |

📌 **一句话：**

> HTTP 是“面向资源”的通信方式，RPC 是“面向函数”的通信方式。

## ⚙️ 二、RPC 工作原理（简化版）

```
Client 调用本地代理对象（Stub）
    ↓ 序列化请求（方法名 + 参数）
    ↓ 通过网络发送给服务端
Server 收到请求 → 反序列化 → 调用本地函数
    ↓ 返回结果序列化并发送回客户端
Client 接收结果 → 反序列化 → 返回调用方
```

📌 就像**本地函数调用**一样简单，隐藏了通信细节。

------

## 🚀 三、为什么要用 RPC？（核心优势）

| 优势                   | 说明                                           |
| ---------------------- | ---------------------------------------------- |
| ✅ **更高性能**         | 使用二进制协议（Protobuf / Thrift）压缩数据量  |
| ✅ **开发体验一致**     | 调用远程服务像调用本地函数                     |
| ✅ **跨语言支持**       | 通过统一 IDL（接口描述语言）生成代码           |
| ✅ **更强可扩展性**     | 天然适合微服务架构（服务注册、发现、负载均衡） |
| ✅ **可嵌入中间层能力** | 支持限流、熔断、追踪、超时控制等机制           |

📌 RPC 是**分布式系统的通信骨架**，HTTP 更多用于外部开放接口。

## 🔧 四、典型 RPC 框架示例

| 框架                     | 序列化方式         | 传输协议 | 特点                           |
| ------------------------ | ------------------ | -------- | ------------------------------ |
| **gRPC**                 | Protobuf           | HTTP/2   | 谷歌开源，跨语言，支持流式传输 |
| **Dubbo**                | Hessian / Protobuf | TCP      | Java 生态广泛                  |
| **Thrift**               | Thrift Binary      | TCP      | 跨语言，轻量快速               |
| **Motan / Tars / GoRPC** | 各自优化           | TCP      | 高性能、内部服务通信           |

------

## 🧩 五、二者关系总结图

```
          ┌──────────────┐
          │   HTTP 协议   │  → 面向资源，通用性强
          └──────┬───────┘
                 ↓
          ┌──────────────┐
          │   RPC 框架    │  → 面向函数，高性能通信
          └──────────────┘
                 ↓
        封装传输协议（TCP/HTTP2）
        + 序列化机制 + 服务治理功能
```

## 🧠 六、速记口诀

> **“HTTP 拿资源，RPC 调函数。”**
>
> - HTTP：跨网访问网页和接口。
> - RPC：微服务内部高效通信。

📌 **一句话总结：**

> RPC 不是替代 HTTP，而是**站在 HTTP 之下做性能和抽象优化的通信协议**。

# “3.9 既然有 HTTP 协议，为什么还要有 WebSocket

## 🌐 一、HTTP 的“先天缺陷”

HTTP 是 **“请求–响应”** 模型：

- **被动通信**：客户端必须先发请求，服务器才能回。
- **无状态**：服务器不记得客户端的连接上下文。
- **单向传输**：服务器无法主动推送数据。

📌 **一句话总结：**

> “HTTP 是单行道，只能客户端先说话。”

## 🚀 二、WebSocket 的出现：让通信“双向”

| 对比项   | **HTTP**           | **WebSocket**                    |
| -------- | ------------------ | -------------------------------- |
| 通信方式 | 单向（请求→响应）  | 双向（全双工）                   |
| 连接状态 | 短连接（每次重建） | 长连接（一次建立持续使用）       |
| 传输协议 | 基于 TCP 的 HTTP   | 基于 TCP，独立协议（ws://）      |
| 消息格式 | 文本请求/响应      | 二进制或文本帧（frame）          |
| 应用场景 | 浏览网页、接口请求 | 实时通信、推送、游戏、股票行情等 |

📌 **一句话记忆：**

> WebSocket = “服务器也能主动说话的 HTTP”。

## ⚙️ 三、握手阶段：从 HTTP 升级而来

1️⃣ **客户端发起 HTTP 请求**（含升级头）：

```
GET /chat HTTP/1.1
Host: server.com
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Key: xxxxx
Sec-WebSocket-Version: 13
```

2️⃣ **服务器同意升级**：

```
HTTP/1.1 101 Switching Protocols
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Accept: yyyyy
```

✅ 握手成功后，连接升级为 **WebSocket 通道**。
 此后通信不再走 HTTP，而是**全双工帧通信**。

## 🧩 四、通信机制

```
HTTP:   Client → Server → Client → Server  (请求-响应)
WebSocket:  ↔ 持久连接，双方可随时发送消息
```

- 服务器可主动推送数据（如消息、心跳、事件）。
- 数据帧格式紧凑，性能比 HTTP 轮询高得多。

------

## ⚙️ 五、WebSocket 的优势

| 优势           | 说明                             |
| -------------- | -------------------------------- |
| ✅ **低延迟**   | 持久连接，无需频繁握手           |
| ✅ **实时性强** | 服务器可主动推送                 |
| ✅ **传输高效** | 二进制帧、小包头                 |
| ✅ **状态保持** | 长连接能维护上下文（如登录状态） |

📌 **HTTP 轮询 → 长轮询 → SSE → WebSocket**
 是前端实时通信的进化路线。

------

## ⚠️ 六、使用场景

- 实时聊天（IM、微信网页版）
- 实时通知 / 推送（GitHub 通知、在线客服）
- 在线游戏 / 实时竞价
- 股票、币价行情
- 协同编辑（Google Docs）

##  七、速记口诀

> **“HTTP 只能问，WebSocket 会聊天。”**
>
> - HTTP：被动、短连接、单向。
> - WebSocket：主动、长连接、双向。

📌 **一句话总结：**

> WebSocket 是 **为“实时性”而生的持久双向通信协议**，
>  让服务器和客户端像“打电话”一样即时对话。
