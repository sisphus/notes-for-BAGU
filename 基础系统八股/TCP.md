# 4.1 TCP 三次握手与四次挥手

## 🚀 一、TCP 三次握手（Three-way Handshake）

### 🔹 目的：

建立**可靠的全双工连接**，同步双方的 **序列号（seq）** 与 **确认号（ack）**。

### 🔹 三步过程：

| 步骤            | 发送方          | 接收方                       | 说明                     |
| --------------- | --------------- | ---------------------------- | ------------------------ |
| **① SYN**       | 客户端 → 服务器 | SYN=1, seq=x                 | 请求建立连接（我想连你） |
| **② SYN + ACK** | 服务器 → 客户端 | SYN=1, ACK=1, seq=y, ack=x+1 | 同意连接并响应           |
| **③ ACK**       | 客户端 → 服务器 | ACK=1, ack=y+1               | 确认连接建立成功         |

📌 **此时：** 双方都已确认“我能发、你能收”。

------

### 🔹 为什么要三次？

- 两次无法确认**双方接收能力正常**；第三次确认确保双方“都准备好”。
   📌 **口诀：**“三次是因为要‘我发你收，我收你发’都通。”

## 🧩 二、TCP 四次挥手（Four-way Handshake）

### 🔹 目的：

安全关闭连接，让双方都能“优雅下线”。

------

### 🔹 四步过程：

| 步骤      | 发送方          | 接收方         | 说明                   |
| --------- | --------------- | -------------- | ---------------------- |
| **① FIN** | 客户端 → 服务器 | FIN=1, seq=u   | 客户端说“我没数据发了” |
| **② ACK** | 服务器 → 客户端 | ACK=1, ack=u+1 | 服务器确认收到         |
| **③ FIN** | 服务器 → 客户端 | FIN=1, seq=v   | 服务器也没数据发了     |
| **④ ACK** | 客户端 → 服务器 | ACK=1, ack=v+1 | 客户端确认关闭         |

📌 之后客户端进入 **TIME_WAIT（2MSL）** 状态等待，以防丢包。

## 🧠 三、经典面试题汇总

| 问题                         | 答案精华                                                |
| ---------------------------- | ------------------------------------------------------- |
| **1️⃣ 为什么握手是三次？**     | 为确保双方“发送 + 接收”能力正常，防止旧连接影响新连接。 |
| **2️⃣ 为什么挥手是四次？**     | TCP 全双工，关闭需双方独立确认（读写通道分开）。        |
| **3️⃣ 为什么要 TIME_WAIT？**   | 确保最后 ACK 能被对方收到、防止旧连接残包干扰新连接。   |
| **4️⃣ TIME_WAIT 过多怎么办？** | 调整内核参数（如 `tcp_tw_reuse`）、负载均衡分摊连接。   |
| **5️⃣ SYN Flood 攻击是什么？** | 攻击者伪造大量 SYN，耗尽服务器半连接队列。              |
| **6️⃣ 怎么防 SYN Flood？**     | SYN Cookies、缩短超时、启用防火墙过滤。                 |

## ⚙️ 四、连接状态总结（重点背诵）

| 阶段  | 客户端状态  | 服务器状态  |
| ----- | ----------- | ----------- |
| 握手1 | SYN_SENT    | LISTEN      |
| 握手2 | SYN_SENT    | SYN_RCVD    |
| 握手3 | ESTABLISHED | ESTABLISHED |
| 挥手1 | FIN_WAIT_1  | CLOSE_WAIT  |
| 挥手2 | FIN_WAIT_2  | LAST_ACK    |
| 挥手3 | TIME_WAIT   | CLOSED      |

# 4.2 TCP 重传机制、滑动窗口、流量控制、拥塞控制

## 🌐 一、TCP 可靠传输的四大机制

> **核心目标：保证数据“不丢、不乱、不错、不爆”。**

| 机制           | 解决问题         | 关键词              |
| -------------- | ---------------- | ------------------- |
| ✅ **重传机制** | 丢包怎么办       | 超时重传 / 快速重传 |
| ✅ **滑动窗口** | 提高传输效率     | 连续发送多包        |
| ✅ **流量控制** | 接收方太慢怎么办 | 接收端窗口调整      |
| ✅ **拥塞控制** | 网络太挤怎么办   | 慢启动 + 拥塞避免等 |

## ⚙️ 二、重传机制（Retransmission）

| 类型                              | 原理                               | 场景             |
| --------------------------------- | ---------------------------------- | ---------------- |
| **1️⃣ 超时重传（RTO）**             | 超时未收到 ACK → 重新发送          | 丢包或超时       |
| **2️⃣ 快速重传（Fast Retransmit）** | 收到连续 3 个重复 ACK → 提前重传   | 网络延迟小但丢包 |
| **3️⃣ SACK 选择确认**               | 告诉发送方哪些包已收到（部分确认） | 提高重传效率     |
| **4️⃣ Duplicate ACK**               | 用来触发快速重传                   | ACK 累计机制     |

📌 **一句话记忆：**“超时等太久，三 ACK 快补救。”

## 🚀 三、滑动窗口（Sliding Window）

> **让 TCP 在收到 ACK 之前能连续发多个包，提高带宽利用率。**

| 概念         | 含义                       |
| ------------ | -------------------------- |
| **发送窗口** | 还能发送但未确认的数据范围 |
| **接收窗口** | 接收方能缓存的数据范围     |
| **滑动机制** | 每次收到 ACK，窗口向前滑动 |

📌 **优点：** 实现流量控制的基础，提高传输并行度。

## ⚙️ 四、流量控制（Flow Control）

> **防止“快发送方”把“慢接收方”压垮。**

- 接收方通过 **ACK 报文中的 `Window` 字段** 告诉发送方自己还能收多少。
- 若接收方忙不过来 → 通告窗口变小甚至为 0。
- 发送方暂停发送，等接收方发 **窗口更新（Zero Window Probe）** 再恢复。

📌 **一句话记忆：**“你说能收多少，我就发多少。”

## 📉 五、拥塞控制（Congestion Control）

> **防止网络因过多数据而“塞车”。**

四个阶段算法（面试必背）👇

| 阶段                                  | 说明                            | 特征         |
| ------------------------------------- | ------------------------------- | ------------ |
| **1️⃣ 慢启动 (Slow Start)**             | cwnd 从 1 MSS 起，每次 ACK 翻倍 | 指数增长     |
| **2️⃣ 拥塞避免 (Congestion Avoidance)** | 到达阈值后线性增长              | 稳定传输     |
| **3️⃣ 快重传 (Fast Retransmit)**        | 3 个重复 ACK → 立即重传         | 提高恢复速度 |
| **4️⃣ 快恢复 (Fast Recovery)**          | cwnd 减半，不回到 1             | 加速恢复阶段 |

📌 **口诀：**“慢启线增三快四不归零。”

## 🧩 六、四者关系图（总结）

```
        ┌──────────────┐
        │  重传机制：保证可靠  │
        ├──────────────┤
        │  滑动窗口：提升效率  │
        ├──────────────┤
        │  流量控制：防止压垮对方 │
        ├──────────────┤
        │  拥塞控制：防止塞满网络 │
        └──────────────┘
```

📌 **一句话总结：**

> TCP 用重传保证可靠性，
>  用滑动窗口提效率，
>  用流量控制防爆仓，
>  用拥塞控制防堵车。

# 4.3 TCP 实战抓包分析

## 🌐 一、抓包工具与目的

| 工具                    | 说明                                                |
| ----------------------- | --------------------------------------------------- |
| **Wireshark / tcpdump** | 常用 TCP 抓包分析工具                               |
| **目的**                | 验证三次握手、四次挥手、数据传输与 ACK 交互是否正常 |

📌 **一句话：**

> 抓包能看见 TCP 协议在真实网络中的“呼吸节奏”。

## ⚙️ 二、三次握手抓包分析

### 抓包序列（Wireshark 典型）

| 序号            | 标志位          | 说明                       |
| --------------- | --------------- | -------------------------- |
| **① SYN**       | 客户端 → 服务器 | 建立连接请求（seq=x）      |
| **② SYN + ACK** | 服务器 → 客户端 | 同意连接（seq=y, ack=x+1） |
| **③ ACK**       | 客户端 → 服务器 | 确认连接（ack=y+1）        |

📌 **观察点：**

- Flags：`SYN`、`ACK`
- TCP State 转变：`SYN_SENT → ESTABLISHED`
- RTT 可从 SYN → SYN+ACK 的时间差估算

## 🧩 三、四次挥手抓包分析

| 序号      | 标志位             | 说明                 |
| --------- | ------------------ | -------------------- |
| **① FIN** | 客户端发送关闭请求 | “我没数据了”         |
| **② ACK** | 服务器确认         | “收到，先等我处理完” |
| **③ FIN** | 服务器关闭发送方向 | “我也没数据了”       |
| **④ ACK** | 客户端确认关闭     | 进入 TIME_WAIT       |

📌 **关键字段：**

- Flags：`FIN`、`ACK`
- 状态流：`FIN_WAIT_1 → FIN_WAIT_2 → TIME_WAIT`

## ⚙️ 四、数据传输抓包分析

### 正常通信时：

- 数据包标志：`[PSH, ACK]`（Push 表示立即发送）
- ACK 确认号递增（累计确认机制）
- TCP 窗口值（Window）动态变化，反映流量控制

📌 **观察指标：**

- `Seq`（序列号）递增
- `Ack`（确认号）连续
- `Win`（窗口大小）波动

------

### 重传场景示例：

| 现象          | 原因         | 标志                      |
| ------------- | ------------ | ------------------------- |
| **超时重传**  | 未收到 ACK   | 相同 Seq 号、时间间隔较长 |
| **快速重传**  | 连续重复 ACK | 相同 Seq 号，ACK 重复三次 |
| **SACK 重传** | 选择性重传   | 抓包可见 SACK 选项字段    |

📌 **分析思路：** 看 Seq 是否重复 + ACK 是否累积。

## ⚙️ 五、流量与拥塞控制观测点

| 指标            | 说明                                       |
| --------------- | ------------------------------------------ |
| **Window Size** | 接收方可接受的数据量（流量控制）           |
| **CWND 变化**   | 拥塞控制：慢启动时指数增长，网络抖动时减半 |
| **RTT / RTO**   | 用于评估网络延迟与超时重传间隔             |

📌 **Tip：** 在 Wireshark 中用 `Statistics → TCP Stream Graphs → Time-Sequence Graph (Stevens)` 查看 cwnd 与 ACK 关系。

------

## 📊 六、TIME_WAIT 与重连分析

- TCP 连接关闭后，客户端保持 **TIME_WAIT（2MSL）**
- 抓包表现：最后两个 ACK → 连接未立即断开
- 若快速重连：可能出现 **端口重用 (tcp_tw_reuse)**

📌 **判断是否重连：** 看 `Port` 是否复用、`Seq` 是否从头开始。

# 4.4 TCP 半连接队列与全连接队列

## 🌐 一、TCP 建立连接的两个队列

在 **三次握手期间**，服务器内核维护两个队列👇

| 队列类型       | 英文名           | 作用                                             | 状态                |
| -------------- | ---------------- | ------------------------------------------------ | ------------------- |
| **半连接队列** | **SYN Queue**    | 存放已收到 **SYN**、尚未完成握手的连接           | 状态：`SYN_RCVD`    |
| **全连接队列** | **Accept Queue** | 存放已完成三次握手、等待应用层 `accept()` 的连接 | 状态：`ESTABLISHED` |

📌 **一句话记忆：**

> “半连接队列等确认，全连接队列等 accept。”

## ⚙️ 二、握手过程中两队列的变化

```
① 收到 SYN → 放入 半连接队列
② 发回 SYN+ACK 等待 ACK
③ 收到客户端 ACK → 从半连接队列移至全连接队列
④ 应用层 accept() → 取出连接交给程序使用
```

## 🧩 三、队列示意图

```
客户端                     服务器
  SYN  ───────────►  [半连接队列]
                     SYN_RCVD
  ACK  ◄───────────  完成握手 → 移入 [全连接队列]
                     ESTABLISHED
  应用层 accept() → 拿出连接使用
```

## ⚠️ 四、常见问题与面试高频考点

### ✅ 1️⃣ 半连接队列满会怎样？

- 新的 **SYN 请求被丢弃** 或触发 **SYN cookies** 机制。
- 可能导致连接延迟或失败。

📌 **内核参数**：

```
net.ipv4.tcp_max_syn_backlog  # 半连接队列大小
net.ipv4.tcp_syncookies = 1   # 启用 SYN Cookies 防攻击
```

### ✅ 2️⃣ 全连接队列满会怎样？

- 三次握手完成，但应用层还没来得及 `accept()`。
- 此时新连接握手成功但会被 **RST 重置**（连接失败）。

📌 **内核参数**：

```
net.core.somaxconn   # 全连接队列最大长度
```

### ✅ 3️⃣ SYN Flood 攻击是什么？

> 攻击者伪造大量 SYN 请求，使半连接队列爆满，服务器资源被占用。

📌 **防御手段：**

- 启用 SYN Cookies（无状态握手验证）
- 限速 / 防火墙 / 负载均衡
- 增大队列：`tcp_max_syn_backlog`

## ⚙️ 五、优化手段汇总

| 优化项                | 说明                                             |
| --------------------- | ------------------------------------------------ |
| `tcp_max_syn_backlog` | 调大半连接队列                                   |
| `somaxconn`           | 调大全连接队列                                   |
| `tcp_syncookies=1`    | 抗 SYN Flood 攻击                                |
| `backlog` 参数        | `listen(fd, backlog)` 控制应用层 accept 队列长度 |

📌 **注意：**
 应用层 backlog ≤ `somaxconn`，最终取两者较小值。

## ✅ 七、总结一句话

> TCP 连接从半连接队列进入全连接队列，
>  是 **从内核确认 → 到应用接管** 的过渡过程。
>  两个队列的管理直接影响服务器**并发性能与抗攻击能力**。
