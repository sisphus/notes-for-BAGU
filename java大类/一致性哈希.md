## 为什么需要一致性哈希？

在分布式缓存、负载均衡、数据分片等场景中，我们经常需要将数据（或请求）均匀地分配到多个服务器节点上。最简单的方法是使用取模哈希（Modulo Hashing）：hash(key) % N，其中 N 是服务器节点的数量。然而，取模哈希有一个致命的缺点：

•当节点数量 N 发生变化时（增加或减少节点），几乎所有的数据都需要重新计算哈希值并重新映射到新的节点上。 这会导致大量的数据迁移，对系统造成巨大的冲击，尤其是在缓存系统中，会导致缓存命中率骤降，进而压垮后端数据库。

一致性哈希就是为了解决这个痛点而诞生的。

## 一致性哈希的核心思想

一致性哈希将整个哈希值空间组织成一个虚拟的圆环（Ring）。

1. 哈希环的构建：

​	•首先，定义一个哈希值空间，例如 0 到 2^32 - 1。

​	•将这个哈希空间首尾相连，形成一个圆环。

2. 节点（服务器）的映射：

​	•将每个服务器节点（例如，根据其 IP 地址或名称）通过哈希函数计算出一个哈希值，并将这个哈希值映射到哈希环上的一个点。

​	•例如，服务器 A 映射到点 H(A)，服务器 B 映射到点 H(B)，服务器 C 映射到点 H(C)。

3. 数据（键）的映射：

​	•将每个数据键（例如，缓存的 key、用户的 ID）也通过相同的哈希函数计算出一个哈希值，并将这个哈希值映射到哈希环上的一个点。

​	•例如，数据 key1 映射到点 H(key1)，数据 key2 映射到点 H(key2)。

4. 数据到节点的分配：

   •对于哈希环上的一个数据点 H(key)，它会被分配到顺时针方向遇到的第一个服务器节点。

   •例如，如果 H(key1) 在 H(A) 和 H(B) 之间，那么 key1 就分配给服务器 B。

## 动态增减节点时的优势

##### •增加节点：

•假设新增一个服务器 D，它被映射到哈希环上的 H(D) 点。

•只有在 H(D) 逆时针方向到 H(C) 之间的那些数据（原来由 C 负责的数据）需要重新分配给 D。

•其他节点（A 和 B）负责的数据不受影响，无需迁移。数据迁移量被最小化。

#### •删除节点：

•假设服务器 B 宕机或被移除。

•原来由 B 负责的数据（在 H(A) 顺时针到 H(B) 之间的所有数据）现在会顺时针移动到下一个节点，即服务器 C。

•只有这部分数据需要重新分配给 C。

•其他节点（A 和 C）负责的数据不受影响，无需迁移。数据迁移量同样被最小化。

## 虚拟节点（Virtual Nodes / Replicas）

虽然一致性哈希解决了数据迁移量大的问题，但如果节点数量较少，或者节点在哈希环上的分布不均匀，可能会导致：

•数据倾斜：某些节点可能负责了过多的数据，而另一些节点负责的数据很少，导致负载不均衡。

#### 为了解决这个问题，引入了虚拟节点的概念：

•每个真实的服务器节点，不再只映射到哈希环上的一个点，而是映射到多个（例如 100 个）不同的虚拟节点。

•这些虚拟节点在哈希环上随机分布，从而使得数据在真实节点上的分布更加均匀。

•当数据分配到某个虚拟节点时，实际上就是分配到了该虚拟节点对应的真实服务器上。

•增加或删除真实节点时，只需要增加或删除其对应的所有虚拟节点，数据迁移量依然保持在较低水平，同时负载均衡效果更好。
