# **Redis 核心知识总结**

### **一、Redis 定义与特点**

- **Redis 是什么：**

  一种 **基于内存（in-memory）** 的高性能键值数据库，读写速度极快。

- **常见用途：**

  - 缓存（Cache）
  - 消息队列（Message Queue）
  - 分布式锁（Distributed Lock）
  - 排行榜、计数器、会话存储等

### **二、Redis 支持的数据类型**

| **类型**               | **说明**                   | **常见使用场景**        |
| ---------------------- | -------------------------- | ----------------------- |
| **String**             | 最基本类型                 | 缓存单值、计数器        |
| **Hash**               | 键值对集合（类似小型对象） | 用户信息、配置表        |
| **List**               | 有序列表                   | 消息队列、任务列表      |
| **Set**                | 无序集合                   | 去重、标签系统          |
| **ZSet（Sorted Set）** | 有序集合（带分数）         | 排行榜、按权重排序      |
| **Bitmaps**            | 位操作                     | 统计登录/打卡等布尔状态 |
| **HyperLogLog**        | 近似去重计数               | UV 统计（独立访问数）   |
| **Geo**                | 地理信息                   | 附近的人/位置查询       |
| **Stream**             | 流数据结构                 | 消息队列、日志流处理    |

### **三、Redis 的主要特性**

- **高性能：** 操作全部在内存中完成。

- **单线程模型：** 所有命令原子执行，无锁竞争。

- **事务支持：** MULTI / EXEC 实现事务语义。

- **持久化：**

  - RDB（快照）
  - AOF（追加日志）
  - 可结合使用。

- **集群模式：**

  - 主从复制（Master-Slave）
  - 哨兵模式（Sentinel）
  - 分片集群（Cluster）

- **扩展功能：**

  发布/订阅（Pub/Sub）、Lua 脚本、内存淘汰机制、过期删除机制等。

## **⚔️Redis vs Memcached 对比总结**

| **对比项**   | **Redis**                               | **Memcached**              |
| ------------ | --------------------------------------- | -------------------------- |
| **数据结构** | 丰富（String、Hash、List、Set、ZSet等） | 仅支持简单的 key-value     |
| **持久化**   | 支持 RDB/AOF，可重启恢复数据            | 不支持，数据断电即失       |
| **集群模式** | 原生支持（主从、哨兵、Cluster）         | 无原生集群，依赖客户端分片 |
| **扩展功能** | 支持事务、Lua 脚本、发布订阅等          | 不支持                     |
| **内存管理** | 可选择多种淘汰策略（LRU、LFU等）        | 采用 LRU 淘汰              |
| **性能**     | 速度略低但功能更强                      | 极致简单，性能略高一点     |
| **使用场景** | 缓存 + 消息队列 + 分布式锁              | 纯缓存使用                 |

# **为什么用 Redis 作为 MySQL 的缓存**

### **一、核心原因：高性能 + 高并发**

Redis 能极大提升数据库系统的整体吞吐量与响应速度。

### **1️⃣ 高性能（Performance）**

- **问题：**

  MySQL 读数据是从**磁盘**读取，速度慢；而 Redis 直接从**内存**中读取，速度极快。

- **解决方案：**

  首次访问数据 → 从 MySQL 查询 → 写入 Redis 缓存

  下次访问相同数据 → 直接读 Redis → 避免磁盘 I/O

- **结果：**

  - 读写延迟从毫秒级降到微秒级
  - 页面加载速度和系统响应时间显著提升

- **补充：**

  数据更新时需保持 **MySQL 与 Redis 的数据一致性（双写一致性问题）**。

  常见方案包括：

  - 延时双删（删除缓存再更新数据库）
  - 异步更新（消息队列同步）
  - 读写穿透检测机制

### **2️⃣ 高并发（Concurrency）**

- **QPS 对比（单机性能）：**

  - Redis：10 万级 QPS（内存操作）
  - MySQL：1 万级 QPS（磁盘操作）

- **原理：**

  Redis 单线程事件循环 + I/O 多路复用，极大减少上下文切换。

  MySQL 受限于事务、磁盘 I/O、锁竞争等，无法支撑超高并发请求。

- **效果：**

  - Redis 承担大部分热点数据访问请求
  - MySQL 负载显著降低
  - 系统整体吞吐量提高 5~10 倍以上

### **🧠 总结一句话：**

> Redis 让「**热点数据走内存**」，MySQL 负责「**持久化存储**」，

> 两者结合形成了典型的 **高性能高并发架构**：

> **前端缓存（Redis） + 后端数据库（MySQL）**。

## **Redis 五种常见数据类型的底层实现（3.x → 7.x 对比精要）**

| **Redis 数据类型**   | **早期版本（3.0，《Redis 设计与实现》）**    | **新版本（7.0）**                    | **说明与核心机制**                                           |
| -------------------- | -------------------------------------------- | ------------------------------------ | ------------------------------------------------------------ |
| **String（字符串）** | **SDS（Simple Dynamic String）**             | ✅ 仍为 SDS                           | 📌 Redis 不使用 C 字符串，而使用 SDS：• 支持二进制安全（可存图片、视频等）• len 字段记录长度 → O(1) 获取长度• 自动扩容，避免缓冲区溢出 |
| **List（列表）**     | 双向链表（linkedlist）或压缩列表（ziplist）  | ✅ **quicklist**（融合结构）          | 📌 quicklist = 多个 ziplist + 双向链表：• 小数据块用 ziplist 节省内存• 大数据块用链表提高插入删除效率 |
| **Hash（哈希）**     | 压缩列表（ziplist）或哈希表（hashtable）     | ✅ **listpack** 或 hashtable          | 📌 3.x 的 ziplist 已废弃，7.x 用 listpack：• 小型 Hash → listpack 紧凑存储• 大型 Hash → 哈希表，O(1) 查找 |
| **Set（集合）**      | 整数集合（intset）或哈希表（hashtable）      | ✅ **intset / hashtable**（机制未变） | 📌 • 全是整数 & 元素少时 → intset（节省内存）• 否则 → hashtable（哈希查找） |
| **ZSet（有序集合）** | 压缩列表（ziplist）或跳表（skiplist + dict） | ✅ **listpack + skiplist + dict**     | 📌 • 小集合 → listpack 紧凑存储• 大集合 → 跳表（按 score 排序）+ dict（按成员查找）配合使用 |

### **🧱 底层数据结构简要说明**

| **数据结构**                     | **用途**            | **特点**                                     |
| -------------------------------- | ------------------- | -------------------------------------------- |
| **SDS（Simple Dynamic String）** | String              | 动态扩容、二进制安全、O(1) 获取长度          |
| **ziplist（压缩列表）**          | 旧版 List/Hash/ZSet | 连续内存存储，节省空间（已废弃）             |
| **listpack（轻量压缩列表）**     | 新版 List/Hash/ZSet | ziplist 的改进版，更高效安全                 |
| **quicklist**                    | List                | 多个 listpack 组成的双向链表，结合空间和性能 |
| **intset（整数集合）**           | Set                 | 仅存整数，紧凑排序结构                       |
| **hashtable（哈希表）**          | Hash/Set            | O(1) 查询、支持 rehash                       |
| **skiplist（跳表）**             | ZSet                | 有序结构，支持范围查找与排序                 |

### **⚙️ Redis 7.0 的优化方向**

1. **压缩列表（ziplist）废弃** → 全部替换为 **listpack**（更高效、更安全）。
2. **统一小对象存储结构**：listpack 被复用在 Hash、ZSet、Stream 等多种类型中。
3. **减少内存碎片化** + **提升访问性能**，同时增强可维护性。

# **💡Redis 线程模型精炼总结**

### **一、Redis 是单线程的吗？**

➡️ **结论：Redis 的主线程是单线程的，但整个 Redis 不是单线程程序。**

| **模块**                    | **是否单线程** | **职责**                                          |
| --------------------------- | -------------- | ------------------------------------------------- |
| **主线程（Main Thread）**   | ✅ 单线程       | 处理客户端请求：接收 → 解析 → 读写数据 → 返回结果 |
| **后台线程（BIO Threads）** | ❌ 多线程       | 异步执行耗时任务（避免阻塞主线程）                |

### **二、Redis 后台线程（BIO）机制**

Redis 在运行时会自动启动多个后台线程来执行“耗时、不影响主线程响应”的任务。

| **线程名称**       | **引入版本** | **功能**                 | **示例命令**                              |
| ------------------ | ------------ | ------------------------ | ----------------------------------------- |
| **BIO_CLOSE_FILE** | 2.6          | 异步关闭文件句柄         | close(fd)                                 |
| **BIO_AOF_FSYNC**  | 2.6          | 异步 AOF 刷盘操作        | fsync(fd)（用于 appendfsync everysec）    |
| **BIO_LAZY_FREE**  | 4.0          | 异步释放内存（lazyfree） | UNLINK key、FLUSHDB ASYNC、FLUSHALL ASYNC |

📌 **注意：**

- DEL 命令在主线程同步删除大 key，会造成卡顿；
- UNLINK 命令异步删除，不阻塞主线程。
- 背后逻辑：主线程将任务放入队列，BIO 线程轮询队列并执行。

### 三、Redis 单线程模型（6.0 之前）

Redis 的“单线程”主要指其 **事件循环模型** —— 所有网络 I/O 与命令执行都在主线程中完成。

#### 🔁 工作流程（基于 epoll 的事件循环）

1. **初始化阶段：**
   - `epoll_create()` 创建事件轮询器
   - `socket()` 创建监听 socket
   - `bind()`、`listen()` 绑定端口
   - `epoll_ctl()` 注册“连接事件”回调函数
2. **主循环阶段：**
   - 调用 `epoll_wait()` 阻塞等待事件到来
   - 根据事件类型执行不同回调：
     - **连接事件**：`accept()` 新连接 → 注册“读事件”
     - **读事件**：`read()` 客户端请求 → 解析命令 → 执行 → 写入响应队列
     - **写事件**：`write()` 将结果返回客户端，若没写完则继续注册写事件

📌 Redis 采用 **I/O 多路复用（epoll/kqueue/evport/select）**，在单线程下同时监听成千上万个连接，事件驱动式地高效响应。

### 四、为什么单线程还这么快？

| 优化点                      | 说明                                     |
| --------------------------- | ---------------------------------------- |
| **内存操作**                | 所有数据在内存中读写，无磁盘 I/O 开销    |
| **I/O 多路复用**            | 单线程同时监听多个连接，事件触发立即处理 |
| **非阻塞 I/O + 小任务设计** | 没有上下文切换成本，不存在线程锁竞争     |
| **高效数据结构**            | SDS、listpack、skiplist 等优化过的结构体 |

### 五、Redis 6.0 之后的演进：多线程 I/O 模型（补充）

- 从 **Redis 6.0** 开始引入 **I/O 多线程**，用于加速网络读写阶段。

- 但 **命令执行仍然是单线程**，以保持原子性和一致性。

- 可通过配置项启用：

  ```
  io-threads-do-reads yes
  io-threads 4
  ```

### 🧠 一句话记忆：

> Redis 的 **核心逻辑（命令执行）始终是单线程**，
>  但通过 **I/O 多路复用 + 异步后台线程**，实现了**高性能、高并发、无锁竞争**的事件驱动模型。

# 💡Redis 单线程为什么还这么快？

### ✅ 性能实测

- 官方基准测试：单线程 Redis 吞吐量可达 **10W+ QPS**。

### ⚙️ 高性能原因

1. **所有操作都在内存中完成**
   - Redis 主要瓶颈不是 CPU，而是 **内存速度和网络带宽**。
   - 内存访问延迟只有微秒级，远快于磁盘 I/O。
2. **高效数据结构**
   - 采用如 SDS、ziplist/listpack、skiplist 等结构，时间复杂度低、空间局部性好。
3. **避免多线程开销**
   - 单线程无锁竞争，无上下文切换，CPU 利用率稳定。
4. **I/O 多路复用机制（epoll/select/kqueue）**
   - 一个线程即可同时监听成千上万个连接。
   - Redis 内部通过事件循环轮询 I/O，就像“一个线程高效处理多个 I/O 流”。

## 💡Redis 6.0 之前为什么只用单线程？

1. **CPU 不是瓶颈**
   - Redis 多数场景受限于 **内存和网络 I/O**，而非 CPU。
   - 官方说明：即使在普通 Linux 系统上，单实例也能处理上百万请求/秒。
2. **可水平扩展**
   - 想利用多核 CPU？可以部署 **多个 Redis 实例**（分片或多节点集群）分摊负载。
3. **单线程更稳定**
   - 维护简单，不存在锁竞争、死锁、线程切换等复杂问题。
   - 保证操作顺序性与原子性，可靠性更高。

📌 **一句话理解：**

> 早期 Redis 的性能瓶颈不在 CPU，而在网络与内存；
>  单线程设计更简单、更可靠，还足以支撑高并发。

## 💡Redis 6.0 之后为什么引入多线程？

### 🎯 背景

- 随着硬件性能提升，**网络 I/O 逐渐成为瓶颈**。
- Redis 6.0 引入 **多线程 I/O 模型** 来提升网络读写性能。

### ⚙️ 原理

- **命令执行仍是单线程**（保持原子性）。
- 但 **网络数据读写（I/O 阶段）** 可以由多个线程并行处理。

### 🧩 配置方法

在 `redis.conf` 中启用多线程 I/O：

```
io-threads-do-reads yes    # 开启读线程
io-threads 4               # 开启 4 个 I/O 线程（主线程 + 3 个子线程）
```

> 建议线程数设置为 CPU 核数的 2/3 ~ 3/4（例如 8 核建议 6 线程）。

### 🚀 性能提升

- 官方测试：开启多线程 I/O 后，网络 I/O 吞吐量提升 **约 1 倍**。
- 默认会创建：
  - **1 个主线程（执行命令）**
  - **3 个 BIO 后台线程**（关闭文件、AOF、lazyfree）
  - **N-1 个 I/O 线程**（用于网络读写）

------

## 🧠 总结一句话

| 阶段         | 模型                    | 优点                         |
| ------------ | ----------------------- | ---------------------------- |
| Redis ≤ 5.x  | 单线程 I/O + 命令执行   | 简洁高效，避免锁竞争         |
| Redis = 6.x+ | 多线程 I/O + 单线程命令 | 提升网络吞吐，保持执行原子性 |

> **Redis 快的关键在于：内存操作 + I/O 多路复用 + 单线程无锁竞争。**
>  **Redis 6.0 起的多线程只是优化网络 I/O，不改变单线程执行模型。**

# 💾 Redis 持久化机制详解

### 一、为什么需要持久化？

Redis 的所有数据默认存储在 **内存中**，速度快但易失。
 为了防止重启或宕机后数据丢失，Redis 提供了 **持久化机制**，
 将内存中的数据保存到磁盘，以便重启后恢复。

### 二、Redis 三种持久化方式

| 持久化方式                  | 核心原理                                   | 特点                                     | 适用场景               |
| --------------------------- | ------------------------------------------ | ---------------------------------------- | ---------------------- |
| **RDB 快照**                | 定期将内存数据“快照”写入磁盘（二进制文件） | 速度快、文件小，但可能丢失最近修改的数据 | 定时备份、大规模冷数据 |
| **AOF 日志**                | 将每条写命令以追加形式写入日志文件         | 数据更完整，可重放恢复；文件较大         | 追求数据安全的场景     |
| **混合持久化（RDB + AOF）** | RDB 保存全量数据，AOF 记录增量修改         | 结合两者优点，恢复速度快                 | Redis 4.0+ 默认推荐    |

### 三、AOF 日志的工作机制

<img src="https://cdn.xiaolincoding.com//mysql/other/6f0ab40396b7fc2c15e6f4487d3a0ad7-20230309232240301.png" alt="img" style="zoom:67%;" />

#### 1️⃣ 写命令执行流程

当客户端发送写命令（如 `SET name xiaolin`）时：

1. Redis 主线程先在 **内存中执行命令**（更新数据）；
2. 再把命令以 Redis 通讯协议格式追加写入 **AOF 缓冲区（server.aof_buf）**；
3. 由系统调用 `write()` 写入到 **page cache（内核缓冲区）**；
4. 最终由内核将数据 **刷入硬盘**。

> ⚙️ AOF 文件保存的是命令本身，而不是数据快照。
>  重启时 Redis 逐条“重放”命令以恢复数据。

------

#### 2️⃣ 为什么是“先执行命令再写日志”？

| 原因                 | 解释                                                         |
| -------------------- | ------------------------------------------------------------ |
| **避免语法错误记录** | 如果先写日志再执行命令，命令出错（语法错误）也会被写入日志，恢复时会报错。 |
| **不阻塞主线程**     | 只有写操作成功后才追加日志，避免写日志的 I/O 延迟影响主线程。 |

📌 **风险：**

- 如果执行成功但日志未写入，突然宕机 → 可能丢失数据。
- 因此 Redis 提供了三种“写回策略”来权衡性能与可靠性。

------

### 四、AOF 写回流程图

```
执行写命令
   ↓
命令追加到 server.aof_buf 缓冲区
   ↓
调用 write() 写入 page cache（内核缓存）
   ↓
等待内核将数据刷盘（fsync）
   ↓
写入硬盘（AOF 文件）
```

### 五、AOF 三种写回策略（appendfsync）

| 策略                 | 写回时机                          | 优点                       | 缺点                 |
| -------------------- | --------------------------------- | -------------------------- | -------------------- |
| **Always**           | 每次命令执行后立即 fsync 写入硬盘 | 可靠性最高，保证数据不丢失 | 性能最差，I/O 压力大 |
| **Everysec（默认）** | 每秒写一次（由后台线程定时执行）  | 性能适中，数据最多丢 1 秒  | 可靠性中等           |
| **No**               | 由操作系统决定写回时机            | 性能最好                   | 宕机可能丢大量数据   |

🔍 实际流程：

- Redis 将命令写入 **AOF 缓冲区（user space）**
- `write()` → 写入 **内核页缓存（page cache）**
- `fsync()` → 将页缓存数据**真正写入磁盘**

------

### 六、性能与可靠性对比

| 维度       | Always   | Everysec        | No             |
| ---------- | -------- | --------------- | -------------- |
| 数据安全性 | ★★★★★    | ★★★★☆           | ★★☆☆☆          |
| 性能       | ★☆☆☆☆    | ★★★★☆           | ★★★★★          |
| 丢失风险   | 几乎为零 | 最多丢 1 秒数据 | 可能丢较多数据 |
| 默认选项   | ❌        | ✅ 默认          | ❌              |

> ✅ 推荐设置：
>  生产环境多数采用 `appendfsync everysec`，兼顾性能与可靠性。

------

### 七、混合持久化（Redis 4.0+）

结合 RDB + AOF 优点：

- 先写 RDB 快照（速度快）；
- 同时将 AOF 日志记录增量更新（保证完整性）；
- 恢复时先加载 RDB，再重放 AOF。

📈 优势：恢复速度提升约 2~3 倍，数据安全性接近纯 AOF。

------

### 🧠 一句话总结：

> Redis 持久化的本质是“在性能与安全之间的权衡”：
>
> - **RDB** 快但可能丢数据；
> - **AOF** 稳但写入频繁；
> - **混合持久化** 两全其美。
>    选 `everysec` 通常是最佳实践。

------

# 💡AOF 重写机制详解（AOF Rewrite）

### 一、为什么需要 AOF 重写？

- **问题：**
   AOF 文件是不断追加命令的日志文件，时间一长会变得越来越大。
  - 文件过大 → **占磁盘空间**
  - 恢复数据时需回放全部命令 → **重启变慢**
- **解决方案：**
   Redis 提供了 **AOF 重写机制**，用于对 AOF 文件进行“压缩”。

📌 **核心思想：**

> 将数据库中**当前最新的数据状态**重新写入新的 AOF 文件，
>  只保留必要命令，删除历史冗余操作。

### 二、AOF 重写的基本原理

| 对比     | 普通 AOF 追加                                         | AOF 重写                                 |
| -------- | ----------------------------------------------------- | ---------------------------------------- |
| 写入内容 | 每次写命令都追加到文件                                | 只记录每个键的最新状态                   |
| 文件体积 | 越写越大                                              | 大幅缩小                                 |
| 恢复速度 | 回放全部命令                                          | 只回放最终状态                           |
| 示例     | `set name xiaolin` → `set name xiaolincoding`（两条） | 只保留最新一条：`set name xiaolincoding` |

------

### 三、AOF 重写的触发时机

AOF 文件过大时触发（由配置项控制）：

```
auto-aof-rewrite-percentage 100
auto-aof-rewrite-min-size 64mb
```

当 AOF 文件增长超过上次重写后文件大小的 100%，且总大小超过 64MB 时，触发重写。

### 四、AOF 重写的执行机制

AOF 重写由 **后台子进程（bgrewriteaof）** 执行，非阻塞主线程。

#### ✅ 优势

1. 主线程继续处理客户端请求，不被阻塞。
2. 子进程使用 **写时复制（Copy-on-Write）**，不影响主线程操作。

### 五、AOF 重写的具体流程

#### 1️⃣ 创建子进程

主线程执行 `bgrewriteaof`：

- 调用 `fork()` 创建 **子进程**；
- 父子进程**共享相同内存数据**（写时复制机制）。

#### 2️⃣ 子进程工作内容

- 遍历当前内存数据库中的所有 key；
- 将每个键值对转为一条命令（如 `SET key value`）；
- 写入新的 AOF 临时文件。

#### 3️⃣ 主线程并行操作

在重写期间，主线程仍然执行客户端命令：

- 新命令会写入两个缓冲区：
  - **AOF 缓冲区（原始）**：用于继续写旧 AOF；
  - **AOF 重写缓冲区**：用于记录重写期间新增命令。

### 六、AOF 重写缓冲区的作用

📌 **问题：**
 重写期间主线程仍有新命令写入，
 如果子进程写完新文件后不包含这些命令 → **数据不一致。**

📌 **解决方案：**
 Redis 在子进程开始重写时创建一个：

> **AOF 重写缓冲区（aof_rewrite_buf）**

主线程执行写命令时，会同时写入：

- 普通 AOF 缓冲区（旧日志）
- AOF 重写缓冲区（供重写文件同步使用）

### 七、重写完成后的合并流程

当子进程完成写入后：

1. 向主线程发送信号，通知“重写完成”；
2. 主线程执行以下操作：
   - 将 **AOF 重写缓冲区** 中的命令追加到新文件末尾；
   - 用新文件替换旧的 AOF 文件。

> ✅ 至此，Redis 拥有一个“精简 + 数据完整”的 AOF 文件。

### 八、整体流程图总结

```
┌────────────┐
│ 主线程     │
│  接收命令  │
│  追加到缓冲│────┐
└────┬───────┘    │
     │fork()      │
     ▼             ▼
┌────────────┐  ┌────────────┐
│  重写子进程 │  │  AOF 缓冲区 │
│  扫描内存写新AOF│  │  记录命令  │
└────┬───────┘  └────┬───────┘
     │信号通知        │
     ▼                ▼
┌──────────────────────┐
│ 主线程合并AOF缓冲内容 │
│ 覆盖旧AOF，完成重写   │
└──────────────────────┘
```

------

### 九、关键点总结

| 阶段           | 机制                         | 说明                         |
| -------------- | ---------------------------- | ---------------------------- |
| **触发重写**   | 自动或手动（`bgrewriteaof`） | 文件超限或人工触发           |
| **执行方式**   | 后台子进程 + 写时复制        | 避免阻塞主线程               |
| **数据一致性** | AOF 重写缓冲区               | 保证重写期间新写入命令不丢失 |
| **结果**       | 生成新的精简 AOF 文件        | 覆盖旧文件，体积变小         |

------

### 🧠 一句话记忆：

> **AOF 重写不是简单压缩文件，而是重新“生成数据库命令”**：
>  子进程扫描内存快照 → 写新 AOF 文件 → 主线程合并新命令 → 替换旧文件。
>  这样 Redis 就能在“不阻塞主线程”的情况下，实现 AOF 文件“瘦身”和“性能优化”。

# 💾 Redis RDB 快照机制详解

### 一、为什么需要 RDB 快照？

- **AOF 持久化** 记录的是“命令”，恢复时需要**重放所有写操作**，速度较慢；
- **RDB 快照** 记录的是“某一时刻的内存数据快照”，
   直接还原数据状态，无需重放命令，恢复速度极快。

📌 **一句话理解：**

> AOF 记录“过程”，RDB 记录“结果”。

### 二、RDB 的工作原理

RDB（Redis Database Backup）会把**某一时刻的全部内存数据**保存为一个 `.rdb` 二进制文件。
 当 Redis 重启时，可以从这个文件中直接加载数据回内存。

📸 类比：
 就像给 Redis 的内存数据拍了一张“全景快照”。

### 三、触发 RDB 快照的方式

#### 1️⃣ 手动触发

| 命令       | 说明                                   |
| ---------- | -------------------------------------- |
| **save**   | 主线程执行生成 RDB（会阻塞）           |
| **bgsave** | 通过 fork 创建子进程生成 RDB（非阻塞） |

> 一般生产环境使用 **bgsave**，因为 save 会阻塞主线程。

#### 2️⃣ 自动触发

在配置文件中设置：

```
save 900 1      # 900秒内至少1次写操作触发
save 300 10     # 300秒内至少10次写操作触发
save 60 10000   # 60秒内至少10000次写操作触发
```

满足任意条件即自动执行 `bgsave`。

### 四、RDB 的特点

| 特性         | 说明                             |
| ------------ | -------------------------------- |
| **快照类型** | 全量快照（每次保存所有数据）     |
| **优点**     | 文件小、恢复快、适合定期备份     |
| **缺点**     | 可能丢失最近一次快照后的数据     |
| **适用场景** | 数据量大但对实时性要求不高的场景 |

### 五、RDB 快照是否阻塞主线程？

- **save**：在主线程执行 → 会阻塞，影响请求处理
- **bgsave**：创建子进程执行 → 不阻塞主线程

> Redis 采用 **bgsave** 默认生成快照。

------

### 六、bgsave 的底层机制：写时复制（Copy-On-Write, COW）

#### 📘 基本原理：

- Redis 调用 `fork()` 创建子进程；
- **父进程（主线程）** 继续处理客户端请求；
- **子进程** 负责把当前内存中的数据写入 RDB 文件。

两者共享同一份物理内存，但通过 **写时复制机制** 保证安全。

#### ⚙️ 写时复制过程详解

1️⃣ **fork() 时**：

- 子进程复制父进程的页表（虚拟地址映射表），
   但不复制物理内存数据。
- 父子进程共享同一份物理数据（只读）。

2️⃣ **主线程有写操作时**：

- 操作的那一页数据会被**复制一份副本**；
- 主线程修改副本；
- 子进程仍能读到旧数据用于快照。

🧠 **效果：**

> 子进程负责“拍照”，父进程负责“继续更新”，互不干扰。

<img src="https://cdn.xiaolincoding.com//mysql/other/c34a9d1f58d602ff1fe8601f7270baa7-20230309232304226.png" alt="img" style="zoom:33%;" />

<img src="https://cdn.xiaolincoding.com//mysql/other/ebd620db8a1af66fbeb8f4d4ef6adc68-20230309232308604.png" alt="img" style="zoom:33%;" />

#### 🧩 图解说明：

```
1. fork() 之后：
   父进程页表 → 物理页（共享）
   子进程页表 → 同一个物理页（共享）

2. 主线程写数据时：
   - 系统复制该页 → 新的物理页
   - 父进程页表更新 → 指向新页
   - 子进程页表仍指向旧页
   => 子进程读旧数据，写入 RDB
```

### 七、完整的 bgsave 执行流程

```
主线程执行 bgsave 命令
      ↓
fork() 创建子进程
      ↓
子进程扫描内存数据
      ↓
将数据序列化写入 RDB 文件
      ↓
写入完成后发送信号通知主线程
      ↓
主线程更新 RDB 文件状态（替换旧文件）
```

------

### 八、RDB 快照与数据修改的并发性

✅ **可以同时执行写操作**
 在 bgsave 期间，Redis 仍能处理写请求，
 因为写操作会触发 **写时复制机制（COW）**，保证快照数据一致。

------

### 九、RDB 与 AOF 的对比总结

| 对比项     | RDB                        | AOF                          |
| ---------- | -------------------------- | ---------------------------- |
| 持久化内容 | 数据快照                   | 写命令日志                   |
| 恢复速度   | 快                         | 慢                           |
| 数据安全性 | 可能丢失最近快照之后的数据 | 可配置（everysec 最多丢1秒） |
| 文件大小   | 小                         | 大                           |
| 性能影响   | bgsave 子进程执行，影响小  | 持续写入磁盘，影响中         |
| 适用场景   | 定期备份、高速恢复         | 实时性要求高的系统           |

------

### 🧠 一句话记忆：

> RDB 是“全量快照”，通过 **fork 子进程 + 写时复制（COW）** 实现非阻塞持久化，
>  能在不影响主线程性能的前提下，快速保存和恢复整个 Redis 数据库。

# 💡Redis 混合持久化机制（Hybrid Persistence）

### 一、为什么需要混合持久化？

Redis 早期有两种持久化方式：

- **RDB（快照）**：
   快速恢复、体积小，但频率不好控制，频率低可能丢数据、频率高影响性能。
- **AOF（日记）**：
   数据更完整，但恢复速度慢（需重放所有命令）。

📌 **矛盾：**

> RDB 恢复快但易丢数据，AOF 数据全但恢复慢。

### 二、混合持久化的目标

在 **Redis 4.0** 中，为了兼顾两者优点，引入了：

> ✅ **混合使用 AOF 日志 + RDB 快照** 的方式，
>  既能快速恢复，又能减少数据丢失。

### 三、混合持久化的工作原理

混合持久化在 **AOF 重写（bgrewriteaof）** 时触发：

1. Redis fork 出 **AOF 重写子进程**；
2. 子进程首先将 **当前内存快照（RDB 格式）** 写入新文件；
3. 然后主线程执行的**新增写命令**会被记录到 AOF 缓冲区；
4. 待重写完成后，子进程会将缓冲区中新增的命令以 **AOF 格式** 追加到文件末尾；
5. 写入完成后，新的混合文件替换旧的 AOF 文件。

📘 **文件结构示意：**

```
AOF 文件内容
 ├── RDB 内容（快照数据）
 └── AOF 内容（重写期间新增命令）
```

🧠 **换句话说：**

> 混合持久化的 AOF 文件 = “RDB 格式的全量数据” + “AOF 格式的增量命令”。

### 四、混合持久化的加载过程（恢复数据时）

当 Redis 重启加载混合持久化文件时：

1. 先加载前半部分的 **RDB 内容**（速度快）；
2. 再执行后半部分的 **AOF 命令日志**（补充增量数据）。

📈 这样既能 **快速启动**，又能 **最大限度减少数据丢失**。

### 五、混合持久化的优点与缺点

| 项目             | 优点                                                         | 缺点                                                         |
| ---------------- | ------------------------------------------------------------ | ------------------------------------------------------------ |
| **优点**         | ✅ 结合 RDB 的快速恢复 + AOF 的数据完整性 ✅ 启动更快、数据更安全 ✅ 减少大规模数据丢失风险 | ❌ 文件体积更大（包含 RDB 部分） ❌ 可读性变差（AOF 文件不再是纯文本） ❌ 兼容性：只能在 Redis 4.0+ 使用 |
| **文件加载速度** | 🚀 明显快于纯 AOF                                             |                                                              |
| **数据完整性**   | 📊 明显优于纯 RDB                                             |                                                              |

### 七、执行流程对比图（简化版）

```
[普通 AOF 重写]
  → fork()
  → 子进程遍历内存，生成纯 AOF 文件（SET/DEL/...）

[混合持久化 AOF 重写]
  → fork()
  → 子进程先写 RDB 数据
  → 主线程新增命令写入缓冲区
  → 子进程追加缓冲区命令到文件尾（AOF 格式）
```

------

### 八、总结对比表

| 机制           | 恢复速度 | 数据完整性   | 文件大小 | 推荐使用     |
| -------------- | -------- | ------------ | -------- | ------------ |
| **RDB**        | ✅ 最快   | ❌ 可能丢数据 | 小       | 周期性备份   |
| **AOF**        | ❌ 较慢   | ✅ 数据最完整 | 大       | 实时性要求高 |
| **混合持久化** | ✅ 快     | ✅ 准确       | 较大     | 生产环境推荐 |

------

### 🧠 一句话记忆：

> **混合持久化 = RDB + AOF**：
>  前半段快照快速恢复，后半段命令保证数据完整。
>  启动快、数据安全、性能稳，是 Redis 4.0 之后的主流方案。

# 💡Redis 集群与高可用机制详解

### 一、为什么 Redis 需要集群？

单台 Redis 虽然速度快，但存在三个核心问题：

1. **单点故障（Single Point of Failure）**：主机宕机数据不可用。
2. **性能瓶颈（I/O、CPU、内存受限）**。
3. **无法水平扩展（scalability poor）**。

因此，Redis 提供了以下高可用方案：

- 主从复制（Master-Slave）
- 哨兵模式（Sentinel）
- 切片集群（Cluster）

## 一、主从复制（Master-Slave Replication）

### 1️⃣ 基本原理

- 一台 Redis 主服务器（Master）负责 **写操作**；
- 多台 Redis 从服务器（Slave）负责 **读操作**；
- 主从之间通过复制机制保持数据一致性。

📘 **核心机制：读写分离**

- 客户端写请求 → 主服务器
- 客户端读请求 → 从服务器

> 所有写操作先在主库执行，再异步同步到从库。

### 2️⃣ 同步流程

1. 从库连接主库，发送 `PSYNC` 命令；
2. 主库执行 **全量同步**（初次连接）：
   - 生成 RDB 快照并发送给从库；
   - 从库加载快照后建立连接；
   - 主库将快照期间产生的写命令（增量）再同步过去；
3. 后续保持 **增量复制**（命令实时传播）。

<img src="https://cdn.xiaolincoding.com//mysql/other/2b7231b6aabb9a9a2e2390ab3a280b2d.png" alt="img" style="zoom:80%;" />



### 3️⃣ 异步复制的特性

Redis 的主从同步是 **异步进行的**：

- 主库执行完写命令后立即返回客户端；
- 异步发送写命令给从库；
- 主库不会等待从库确认结果。

📌 **结果：**

> 无法实现强一致性（可能存在短暂延迟或数据丢失）。
>
> Redis 追求的是 **最终一致性**（eventual consistency）。

### 4️⃣ 主从复制的优点与问题

| 优点                   | 缺点                     |
| ---------------------- | ------------------------ |
| ✅ 读写分离，提高性能   | ❌ 主节点宕机后需人工切换 |
| ✅ 数据冗余，提高可靠性 | ❌ 无自动故障转移机制     |
| ✅ 提高读请求吞吐量     | ❌ 主从数据可能延迟不同步 |

------

## 二、哨兵模式（Sentinel）

### 1️⃣ 设计背景

主从复制解决了读写分离问题，但：

> 若主节点宕机，需手动切换，系统不可用。

为此，Redis 提供了 **哨兵模式（Redis Sentinel）**，实现自动故障转移。

### 2️⃣ 哨兵模式架构

#### 架构组成：

- **主节点（Master）**：处理写请求；
- **从节点（Slaves）**：备份主节点数据；
- **哨兵节点（Sentinels）**：监控主从状态，自动故障转移。

📊 **架构示意：**

<img src="https://cdn.xiaolincoding.com//mysql/other/26f88373d8454682b9e0c1d4fd1611b4.png" alt="img" style="zoom:67%;" />

### 3️⃣ 哨兵的核心功能



| 功能                          | 说明                                     |
| ----------------------------- | ---------------------------------------- |
| **监控（Monitoring）**        | 持续 PING 主从节点，检测是否宕机         |
| **通知（Notification）**      | 当节点异常时，通知管理员或其他哨兵       |
| **自动故障转移（Failover）**  | 当主节点宕机时，从节点被选举为新的主节点 |
| **配置更新（Configuration）** | 通知客户端新的主节点地址                 |

------

### 4️⃣ 故障转移过程

1️⃣ 哨兵节点检测到主节点无响应（PING 超时）
 2️⃣ 经过多数哨兵确认（quorum 达成）后判定主节点下线
 3️⃣ 从节点中选举一个新的主节点（通常优先级高、数据最新者）
 4️⃣ 通知其他从节点重新复制新的主节点
 5️⃣ 通知客户端更新主节点地址

📌 Redis 故障转移具备 **自动性 + 高可用性**。

### 5️⃣ 哨兵模式的优点与缺点

| 优点                   | 缺点                           |
| ---------------------- | ------------------------------ |
| ✅ 实现主从自动故障转移 | ❌ 配置复杂，监控节点需独立部署 |
| ✅ 提升高可用性（HA）   | ❌ 切换期间有短暂不可用窗口     |
| ✅ 多哨兵协同判定更可靠 | ❌ 不支持分片（仍为单主架构）   |

## 三、主从 + 哨兵 的协作机制总结

| 模块         | 作用                   | 特点                             |
| ------------ | ---------------------- | -------------------------------- |
| **主从复制** | 读写分离、数据冗余     | 异步复制、需人工切换             |
| **哨兵模式** | 自动故障转移、主从监控 | 自动选主、保证高可用             |
| **结合使用** | 企业主流方案           | 哨兵负责监控和选举，主从负责同步 |

------

### 🧠 一句话记忆：

> Redis 高可用的核心是：
>  **主从复制保证数据备份，哨兵模式保证自动切换。**
>
> - 主从：读写分离
> - 哨兵：监控与选主
> - 两者结合：高可用（HA）架构

## 💡Redis 切片集群模式（Redis Cluster）

------

### 一、为什么需要切片集群（Cluster）？

- 单台 Redis 受限于：
  - **内存容量限制**（一般最多几百 GB）；
  - **网络带宽和 CPU 性能瓶颈**；
  - **单点故障风险**。

📌 解决方案：

> Redis Cluster 通过 **数据分片（Sharding）**
>  将数据分布到多个节点上，实现 **水平扩展（Horizontal Scaling）** 和 **高可用性（HA）**。

------

### 二、Redis Cluster 的核心思想：哈希槽（Hash Slot）

Redis Cluster 并非简单使用一致性哈希，而是采用 **16384 个固定槽（Hash Slot）** 作为数据分片的最小单位。

#### ⚙️ 工作流程：

1. Redis 先定义固定数量的哈希槽（16384 个，编号 0 ~ 16383）；

2. 每个 **键（key）** 通过 CRC16 算法计算哈希值；

3. 对结果取模：

   ```
   slot = CRC16(key) % 16384
   ```

4. 每个哈希槽被分配给一个具体的 Redis 节点。

📘 也就是说：

> **key → slot → 节点**
>  每个 key 都会被映射到一个特定节点上。

### 三、槽与节点的分配方式

#### 方式 ①：自动平均分配

- 当你创建集群时，Redis 自动将 16384 个槽平均分配给各个节点。
- 举例：9 个节点，每个节点约分配 `16384 / 9 ≈ 1820` 个槽。

#### 方式 ②：手动分配

可以通过命令指定槽范围给某个节点：

```
redis-cli -h 192.168.1.10 -p 6379 cluster addslots 0 1
redis-cli -h 192.168.1.11 -p 6379 cluster addslots 2 3
```

### 四、哈希槽映射示例

假设：

- 集群中有 2 个节点；
- 一共 4 个哈希槽（Slot 0~3）。

| 键名 | CRC16 结果    | 取模结果 | 对应槽 | 归属节点     |
| ---- | ------------- | -------- | ------ | ------------ |
| key1 | CRC16(key1)=1 | 1        | Slot 1 | Redis 实例 1 |
| key2 | CRC16(key2)=2 | 2        | Slot 2 | Redis 实例 2 |

📊 图示关系：

```
CRC16(key1)%4 = 1  → Slot 1 → Redis 实例 1
CRC16(key2)%4 = 2  → Slot 2 → Redis 实例 2
```

### 五、Redis Cluster 的数据读写过程

1️⃣ **写入请求：**

- 客户端计算 key 的哈希槽；
- 根据槽所属节点，将请求发送到该节点；
- 若请求发错节点，节点会返回 **MOVED 重定向**，告诉客户端正确节点位置。

2️⃣ **读取请求：**

- 同样根据 key 的哈希槽找到节点；
- 直接从节点读取数据。

> 💡 客户端一般内置槽映射表（slot map），避免频繁重定向。

------

### 六、Redis Cluster 的优势

| 优点               | 说明                                                    |
| ------------------ | ------------------------------------------------------- |
| ✅ **水平扩展**     | 通过增加节点即可线性扩容                                |
| ✅ **高可用性**     | 每个主节点（master）都可配置从节点（slave）进行数据冗余 |
| ✅ **自动故障转移** | 某个主节点宕机时，从节点可自动提升为主节点              |
| ✅ **去中心化设计** | 无单点问题，节点间通过 Gossip 协议通信                  |

------

### 七、Redis Cluster 的注意点与限制

| 限制                    | 说明                                                         |
| ----------------------- | ------------------------------------------------------------ |
| ❌ 不支持多 key 跨槽操作 | 除非所有 key 属于同一哈希槽（可用 hash tag，如 `{user1}.name`、`{user1}.age`） |
| ❌ 无法保证强一致性      | 集群复制是异步的（与主从类似）                               |
| ❌ 槽必须分配完整        | 16384 个槽若有未分配，集群无法正常工作                       |

### 八、Redis Cluster 架构简图

```
          ┌────────────┐
          │ Redis 节点1 │  (槽 0~5460)
          └──────┬─────┘
                 │
                 │
          ┌──────┴─────┐
          │ Redis 节点2 │  (槽 5461~10922)
          └──────┬─────┘
                 │
                 │
          ┌──────┴─────┐
          │ Redis 节点3 │  (槽 10923~16383)
          └────────────┘
```

客户端根据 key 的槽号自动选择对应节点进行操作。

------

### 九、总结对比：主从 vs 哨兵 vs 集群

| 模式                    | 核心功能            | 是否分片 | 高可用性 | 应用场景             |
| ----------------------- | ------------------- | -------- | -------- | -------------------- |
| **主从复制**            | 数据备份 + 读写分离 | ❌        | 部分     | 小规模应用           |
| **哨兵模式**            | 自动故障转移        | ❌        | ✅        | 高可用读写场景       |
| **切片集群（Cluster）** | 数据分片 + 自动选主 | ✅        | ✅        | 海量数据、高并发系统 |

------

### 🧠 一句话记忆：

> Redis Cluster 通过 **16384 个哈希槽** 实现数据分片，
>  每个 key 经过 `CRC16(key) % 16384` 映射到具体节点，
>  既能横向扩展，又能保证高可用，是 Redis 的**分布式核心架构**。
