# 理解这些序列化框架 **底层是怎么把 Java 对象变成字节流（byte[]），然后再还原成对象的。**

## 🚩一、JDK 原生序列化 — 反射 + 对象图遍历 + 魔数头

### ✳️ 原理

JDK 原生序列化依赖 `ObjectOutputStream` / `ObjectInputStream`。
 它利用 **Java 反射机制 + Serializable 接口**，把对象及其引用图中的所有对象递归地写入字节流。

### 🔧 底层流程

1. 调用 `ObjectOutputStream.writeObject(obj)`；
2. 写入 **魔数 header（AC ED 00 05）**，标记这是 Java 的序列化流；
3. 写入类描述信息（`ClassDescriptor`）：类名、serialVersionUID、字段类型等；
4. 依次序列化每个字段的值（通过反射访问私有字段）；
5. 如果字段是对象，再递归调用 `writeObject()`；
6. 最终生成一个二进制流。

反序列化时：

- 读取魔数；
- 加载对应类；
- 根据类描述信息逐个字段还原对象。

### ⚙️ 特点

- 保留完整的对象结构（含类名、类型信息）；
- 可自动处理循环引用；
- 但由于带了大量元信息，**体积大，性能低**。

### 🧠 关键代码

```
ByteArrayOutputStream bos = new ByteArrayOutputStream();
ObjectOutputStream oos = new ObjectOutputStream(bos);
oos.writeObject(user);
byte[] bytes = bos.toByteArray();
```

### 🔩 RPC 层使用方式

JDK 序列化直接写入 Socket 流，但生成的字节流含类元数据，因此：

- **跨语言无法解析**
- 序列化头部大（影响网络带宽）

------

## 🚩二、JSON — 文本化 + 反射 + 动态解析

### ✳️ 原理

JSON 是**文本序列化**方案，它不关心字节结构，而是**把对象转为字符串表示**。

底层原理就是：

1. 通过反射获取所有字段；
2. 把字段名和值拼接成 JSON 格式字符串；
3. 用 UTF-8 编码成字节。

反序列化时：

1. 解析 JSON 文本；
2. 按 key-value 找到字段；
3. 通过反射 `setField` 填回对象。

### ⚙️ 典型实现

- Jackson、Gson、Fastjson 等库；
- 都使用 Java 反射；
- 一般会维护一个 `FieldAccessorCache` 加快访问速度。

### 🧠 示例

```
String json = JSON.toJSONString(user);
User user2 = JSON.parseObject(json, User.class);
```

### ⚙️ 特点

- 人类可读；
- 无需 schema；
- 慢、体积大；
- 丢失类型信息（需要额外字段区分子类）。

------

## 🚩三、Hessian — 二进制结构化协议（动态类型 + tag 标记）

### ✳️ 原理

Hessian 是一个**自描述的二进制协议**，它通过紧凑的**type tag + value**格式表示对象结构。
 每种类型用一个 **单字节的 type code（标志位）** 表示。

例如：

| 类型   | tag(byte) | 描述                           |
| ------ | --------- | ------------------------------ |
| int    | 'I'       | 后跟4字节整数                  |
| string | 'S'       | 后跟长度+UTF-8字符串           |
| object | 'O'       | 后跟类名 + 字段数量 + 字段键值 |

### 🔧 流程（序列化）

1. 写入 type 标识（例如 “O” 表示对象）；
2. 写入类名；
3. 写入字段数；
4. 对每个字段，写入 key + type + value；
5. 对象引用表可重用（防止重复对象重复写入）。

反序列化：

- 解析 type；
- 根据 tag 动态反射创建实例；
- 逐个字段还原。

### 🧠 示例（字节流结构）

```
O 0x00 0x04 User
S name S Bob
I age 25
Z  // end
```

### ⚙️ 特点

- 无需事先 schema；
- 可跨语言；
- 比 JSON 小；
- 比 JDK 快；
- 保留类名信息；
- 实现复杂（依靠字节协议 + Map 类型缓存）。

### 🧩 RPC 框架中的角色

Dubbo 默认序列化方式。
 通信流程：

```
Java对象 → Hessian 二进制 → TCP传输 → Hessian反序列化 → Java对象
```

------

## 🚩四、Kryo — 注册类ID + 高速内存写入 + Unsafe 反射访问

### ✳️ 原理

Kryo 是一个**高性能 Java 二进制序列化库**。
 核心思想是通过 **注册类 ID** 代替类名字符串，从而极大减少字节长度，并使用 **Unsafe + ASM 直接操作对象内存**。

### 🔧 流程

1. 注册类，Kryo 为其分配一个整数 ID；
2. 序列化时：
   - 写入类 ID；
   - 顺序写入字段值；
   - 递归序列化引用对象；
3. 使用 `Output` 类直接写入二进制（ByteBuffer）；
4. 反序列化时读取类 ID，直接从缓存中找到类，创建对象并填充字段。

### ⚙️ 优化机制

- 支持对象引用图；
- 缓存类结构；
- 可通过 ASM 动态生成序列化器；
- 线程不安全（需使用 `KryoPool`）。

### 🧠 示例

```
Kryo kryo = new Kryo();
kryo.register(User.class);
Output output = new Output(new ByteArrayOutputStream());
kryo.writeObject(output, user);
```

### ⚙️ 特点

- 速度极快（靠 direct memory 操作）；
- 序列化体积小；
- 无需反射（可预编译 Schema）；
- 不跨语言；

## 🚩五、Protostuff（基于 Protobuf）— Schema 驱动 + 字段编号压缩

### ✳️ 原理

Protostuff 是对 Google **Protocol Buffers（Protobuf）** 的扩展。
 Protobuf 的核心思想是：
 👉 “不存字段名，只存字段编号”，
 👉 “字段类型用二进制 tag 表示”，
 👉 “字段不存在就跳过”，从而达到最小化。

Protostuff 在此基础上：

- 自动生成 schema（不需要 `.proto` 文件）；
- 利用反射动态推断类结构。

### 🔧 流程

1. 为类构建 Schema（每个字段分配一个编号）；
2. 写入 tag（field number + wire type）；
3. 写入字段值；
4. 反序列化时根据 tag 找对应字段号，填充值。

### ⚙️ 示例（伪代码）

```
Schema<User> schema = RuntimeSchema.getSchema(User.class);
byte[] data = ProtostuffIOUtil.toByteArray(user, schema, LinkedBuffer.allocate());
User u = new User();
ProtostuffIOUtil.mergeFrom(data, u, schema);
```

### 🧠 字节结构示例

```
0A 03 42 6F 62 10 19
=> field(1)=name="Bob", field(2)=age=25
```

### ⚙️ 特点

- 无字段名，极致压缩；
- 超高性能；
- 支持版本演化（字段号不变）；
- 不支持循环引用；
- 不跨语言（Protostuff本身仅 Java）。

| 序列化方式     | 底层机制                      | 是否跨语言 | 性能 | 压缩率 | 是否含类元信息 | 典型框架               |
| -------------- | ----------------------------- | ---------- | ---- | ------ | -------------- | ---------------------- |
| **JDK**        | 反射 + 对象图 + 类描述符      | 否         | ❌ 慢 | ❌ 大   | ✅ 包含         | Java原生、RMI          |
| **JSON**       | 反射 + 文本键值               | ✅          | ❌    | ❌      | ✅              | REST接口、Web通信      |
| **Hessian**    | 二进制自描述协议（tag-based） | ✅          | ✅    | ✅      | ✅              | Dubbo                  |
| **Kryo**       | 类ID + Unsafe + ASM           | 否         | 🚀🚀   | 🚀      | ❌（需注册）    | Spark、Storm           |
| **Protostuff** | Schema 编码（字段号+值）      | ⚠️（部分）  | 🚀🚀🚀  | 🚀🚀🚀    | ❌              | 高性能 RPC、自定义协议 |

如果你是在写自己的 **RPC 框架序列化层**：

✅ 推荐实现顺序：

1. **先实现 JSON 序列化器（便于调试）**；
2. **再实现 Kryo 或 Protostuff 序列化器（追求性能）**；
3. **提供 SPI 插拔机制**（用户可选不同实现）。

# 服务限流（Rate Limiting）八股总结

## 一、为什么要限流？（面试切入点）

> 在分布式系统或 RPC 框架中，当请求量突然激增（例如秒杀、抢票、高并发接口），
>  某个服务节点的 CPU、线程池或连接池可能被打满，导致系统崩溃。
>  **限流（Rate Limiting）** 是一种**主动保护机制**，通过控制单位时间内的请求数，
>  保证系统在可承受范围内稳定运行，从而“保护自己不被压垮”。

✅ **一句话回答**：

> 当服务节点负载过高时，通过限流拒绝部分请求，让系统在可控速率下工作，保证核心服务可用。

## 二、限流的常见算法（核心理论）

| 算法           | 思想                             | 优点                           | 缺点             | 是否支持突发流量 |
| -------------- | -------------------------------- | ------------------------------ | ---------------- | ---------------- |
| 计数器法       | 固定时间窗口计数                 | 实现简单                       | 边界误差大       | ❌                |
| 滑动窗口       | 时间窗口细分滑动统计             | 精度高                         | 实现复杂         | ❌                |
| 漏桶算法       | 固定速率“出水”                   | 控制平均速率稳定               | 无法应对突发流量 | ❌                |
| **令牌桶算法** | 按固定速率产生令牌，请求消耗令牌 | 既能控制平均速率，又能处理突发 | 实现稍复杂       | ✅                |

> 在实际项目中（如 Sentinel、Guava RateLimiter、Dubbo RPC 框架），
>  **令牌桶算法是使用最广的限流算法**。

## 三、令牌桶算法原理（重点要讲）

### 🎯 基本思想

- 系统维护一个**桶（容器）**，最多能装 `CAPACITY` 个令牌；
- 系统**以固定速率（RATE）**往桶里放令牌；
- 每个请求想要被处理，必须**先拿到一个令牌**；
- 若桶空了，请求就会被拒绝（或进入降级逻辑）。

### 🧩 控制两个速率

- **平均速率**：随着时间推移，整体请求处理速率接近 `r`（每秒生成令牌数）。
- **瞬时速率**：突发流量时，最多能同时处理 `n`（桶容量）个请求。

### ⚙️ 与漏桶算法对比

| 特点         | 漏桶 | 令牌桶         |
| ------------ | ---- | -------------- |
| 平均速率控制 | ✅    | ✅              |
| 突发流量支持 | ❌    | ✅              |
| 出口速率     | 固定 | 有上限但可弹性 |

## 五、面试进阶回答模板 🧠

> **面试官问：**「如果 RPC 服务端压力太大，你会怎么保护它？」

✅ **标准回答结构**：

1️⃣ 我会在服务端增加限流机制，防止节点被高并发压垮。
 2️⃣ 实现思路是在执行业务逻辑之前，通过限流算法判断请求是否被允许。
 3️⃣ 常见的限流算法有计数器、滑动窗口、漏桶和令牌桶，我会选用**令牌桶算法**，因为它能同时控制平均速率和瞬时速率。
 4️⃣ 我会为每个接口维护独立的限流器（`RateLimitProvider`），保证接口级限流。
 5️⃣ 当令牌不足时，服务端会直接返回限流响应，不执行业务逻辑，从而实现快速失败。
 6️⃣ 令牌桶实现包含令牌生成速率、最大容量、当前容量与时间戳，线程安全地控制请求通行。
 7️⃣ 这样既能保护系统稳定，又能在突发流量下平滑处理请求。

# 🚀 服务熔断（Circuit Breaker）八股总结

## 一、为什么要熔断？（面试切入点）

> 限流解决了“服务端被打挂”的问题，但在分布式系统中，**调用端也可能被拖垮**。
>  假设：
>
> - 服务 **A → B → C**；
> - 当 C 超时或宕机时，B 调用 C 的线程会长时间等待；
> - A 不断请求 B，B 的线程堆积越来越多；
> - 最终 **C 的异常扩散导致 B、A 全部崩溃**。

这就是**级联失败（Cascade Failure）**。
 因此，**调用端也必须具备自我保护机制**——这就是**熔断（Circuit Breaker）**。

✅ **一句话回答：**

> 熔断是调用端的自我保护机制，当被调用方持续异常时，调用方会暂时停止调用，快速失败，防止级联雪崩。

## 二、熔断设计理念（核心思想）

熔断机制源自电路系统中的**熔断器（Circuit Breaker）**，用于防止电流过大烧毁设备。
 在微服务架构中，熔断器用于防止“请求风暴”扩大化。

### 三种核心状态

| 状态                  | 含义         | 行为                                                         |
| --------------------- | ------------ | ------------------------------------------------------------ |
| **Closed（关闭）**    | 正常状态     | 所有请求正常通过；持续失败计数，当失败比例超过阈值后切换为“打开” |
| **Open（打开）**      | 故障状态     | 所有请求立即失败（快速返回），防止继续压垮下游服务           |
| **Half-Open（半开）** | 探测恢复状态 | 允许部分请求通过，如果成功率达到阈值则恢复为“关闭”，否则重新“打开” |

### 🧠 熔断思想一句话：

> 熔断就是“服务出问题时拉闸断电，过一段时间再试试看能不能恢复”。

## 三、熔断机制工作原理

### 📊 1. 熔断判断逻辑

- 在固定窗口时间内统计失败次数或失败率；
- 若失败率 > 阈值（如 50%），触发熔断；
- 熔断进入“打开”状态后，不再发送真实请求；
- 定时进入“半开”状态试探恢复。

### 🔁 2. 熔断恢复逻辑

- 熔断打开一段时间后（如 10 秒）；
- 允许部分请求通过（比如 10%）；
- 如果这些请求全部成功，则关闭熔断；
- 否则重新打开，等待下一次检测周期。

## 五、面试答题模板 🧠

> **面试官问：**
>  “如果下游服务异常导致调用链级联失败，你怎么保护上游服务？”

✅ **标准回答结构：**

1️⃣ 我会在调用端实现**熔断机制（Circuit Breaker）**，当下游服务持续失败时，调用端主动停止请求。
 2️⃣ 熔断器通常有三种状态：Closed、Open、Half-Open。

Closed：正常请求，通过计数监控失败率；

Open：失败率过高后，直接拒绝所有请求（快速失败）；

Half-Open：在等待一段时间后，允许部分请求尝试恢复。

 3️⃣ 如果半开状态下请求成功率达到阈值，熔断器会重新闭合，否则继续打开。
 4️⃣ 熔断能避免“级联雪崩”，保护调用端自身稳定。
 5️⃣ 在我的项目中，我通过 `CircuitBreakerProvider` 为每个服务维护独立熔断器，并在 `ClientProxy` 中判断是否允许请求，通过记录成功和失败来动态调整状态。
